# World Chain Devnet

The current local devnet path is the native Rust harness in `crates/devnet`, exposed through
`xtask` and the root `just devnet` recipe. The older Kurtosis devnet under `pkg/devnet` is still
kept for reference and fallback, but it is no longer the preferred local workflow.

## Native Rust Devnet

Start the HA devnet:

```bash
just devnet up
```

Run it in the background and stop it later:

```bash
just devnet up -d
just devnet down
```

`just devnet up` builds the native `world-chain` binary and then runs:

```bash
cargo run -p xtask -- devnet up
```

The default preset is `ha-sequencer`. It uses dynamic host ports, prints endpoint URLs at startup,
and keeps running until Ctrl-C. Stable ports are available for manual workflows:

```bash
just devnet up --stable-ports
```

Useful flags:

```bash
just devnet up --print-topology
just devnet up --latest-hardfork tropo
just devnet up --enable-hardfork jovian
just devnet up --disable-hardfork jovian
just devnet up --sequencers 3
just devnet up --block-time-ms 2000
just devnet up --op-challenger
just devnet up --no-observability
```

Lower-level presets remain available through `xtask` for focused testing:

```bash
cargo run -p xtask -- devnet up --preset direct-sequencer
cargo run -p xtask -- devnet up --preset minimal
cargo run -p xtask -- devnet up --preset ha-sequencer --print-topology
```

## Components

The default HA preset starts:

- Anvil-backed L1 dev chain.
- `op-deployer`, used at startup to generate OP contract state, L1 addresses, L2 genesis, and rollup config.
- Three native World Chain execution nodes, each running the local `world-chain` binary.
- Three `op-node` containers, one per execution node.
- Three `op-conductor` containers with a local raft cluster.
- `op-batcher`.
- `op-proposer`.
- Prometheus.
- Grafana with World Chain flashblocks dashboards provisioned.

All native presets sequence directly with flashblocks enabled by default. The new path intentionally
does not wire rollup-boost or tx-proxy. PBH is disabled for native World Chain execution nodes with
zero reserved PBH blockspace and an undeployed sentinel PBH entrypoint.

Because rollup-boost is not present, local flashblocks use the node's dev-only override authorizer
and force-publish mode. `op-node` still drives normal Engine API payload jobs, while the native
World Chain EL self-authorizes flashblock publication for the active local sequencer.

World contract deployment is intentionally deferred. `FeeEscrow`, `FeeRecipient`, PBH contracts,
rollup-boost, tx-proxy, and rundler are not part of the native default HA path.

`op-challenger` is disabled by default because the native devnet does not yet generate the Cannon
prestates required to play local permissioned dispute games. Enable it explicitly with
`just devnet up --op-challenger` when working on challenger/prestate integration.

## Batcher And Proposer Timing

The native HA preset uses short local intervals so L2 data is submitted and output proposals appear
quickly:

```text
op-batcher --poll-interval=1s
op-batcher --max-channel-duration=4
op-batcher --sub-safety-margin=0
op-batcher --num-confirmations=1
op-proposer --game-type=1
op-proposer --proposal-interval=6s
op-proposer --poll-interval=1s
op-proposer --allow-non-finalized
```

`--max-channel-duration=4` is in L1 blocks. With the default 2s L1 block time, the batcher should
close and submit channels roughly every 8 seconds, allowing op-node safe heads to move beyond
genesis and the proposer to submit non-genesis outputs. `--game-type=1` matches the permissioned
dispute game implementation generated by the local `op-deployer` deployment.

## Logs

Foreground runs write logs to stdout and to:

```text
target/devnet/logs/devnet.log
```

Starting the devnet removes the previous log file first, so each `just devnet up` run starts with
a fresh log.

Detached runs write to the same file only. Override the path with:

```bash
WORLD_CHAIN_DEVNET_LOG_FILE=/tmp/world-devnet.log just devnet up
```

The background PID file defaults to:

```text
target/devnet/devnet.pid
```

Override it with `WORLD_CHAIN_DEVNET_PID_FILE`.

## Endpoint File

Each successful startup writes all public endpoints to:

```text
target/devnet/endpoints.json
```

Override it with:

```bash
WORLD_CHAIN_DEVNET_ENDPOINTS_FILE=/tmp/world-devnet-endpoints.json just devnet up
```

The file includes primary URLs such as `sequencer_rpc_url`, `flashblocks_url`, `prometheus_url`,
and `grafana_url`, plus per-component endpoints for each EL, op-node, conductor, batcher,
proposer, Prometheus, and Grafana service. If `--op-challenger` is enabled, its endpoint is included
too.

Process logs are normalized by the devnet wrapper:

- Timestamps from child processes are stripped.
- `level=...` and `lvl=...` fields are stripped after being used as the actual log level.
- stdout renders the service label in bold cyan.
- the log file uses the same leading label without ANSI escapes.

Example stdout shape:

```text
WARN  op-deployer apply msg="Using dev L1 genesis without any customization"
INFO  world-chain-el-0 message="Block added to canonical chain" number=12
INFO  op-node-2 msg="Sync progress" reason="new chain head block"
```

The tracing target remains service-scoped, so filters still work:

```bash
RUST_LOG=op_conductor=error just devnet up
RUST_LOG=op_node=info,world_chain_el=warn just devnet up
```

## Data Directories

Each run creates a lifecycle-owned temp workdir with prefix `world-devnet-op-`, usually under the
platform temp directory. On macOS it looks like:

```text
/var/folders/.../T/world-devnet-op-XXXXXX
```

The startup logs include the exact `workdir=...` path when OP deployment artifacts are generated.

Important paths inside that workdir:

```text
genesis.json
rollup.json
l1-genesis.json
l1-addresses.json
jwt.hex
l2data-0/db
l2data-0/static_files
l2data-1/db
l2data-1/static_files
l2data-2/db
l2data-2/static_files
conductor-0
conductor-1
conductor-2
prestates
```

The `l2data-N` directories are the native World Chain execution node databases. The conductor
directories hold local raft state. The workdir is deleted when the devnet process exits cleanly.

Anvil, Prometheus, Grafana, and op-node peer stores are lifecycle-owned container or memory-backed
state for this harness and are not persisted as stable repo directories.

## Grafana

The HA preset starts Prometheus and Grafana by default. Startup output prints the Grafana URL and
all component endpoint URLs, including the primary sequencer RPC port.

Grafana provisions the Prometheus datasource and imports these dashboards into the
`World Chain Devnet` folder:

```text
pkg/devnet/grafana/dashboards/flashblocks-payload-builder.json
pkg/devnet/grafana/dashboards/flashblocks-validation-pipeline.json
pkg/devnet/grafana/dashboards/flashblocks-p2p.json
reth-overview.json
```

Dashboard coverage:

- `flashblocks-payload-builder.json` for `reth_flashblocks_payload_build_*`.
- `flashblocks-validation-pipeline.json` for `reth_flashblocks_validation_*`.
- `flashblocks-p2p.json` for `reth_flashblocks_p2p*` P2P metrics, including peer-scoped `peer_id` series.
- `reth-overview.json` is downloaded at startup from the upstream Reth dashboard in
  `paradigmxyz/reth` and minimally patched for the devnet Prometheus datasource.

The devnet does not rewrite dashboard queries or force a default node selection. Prometheus assigns
stable `instance` labels for World Chain EL metrics, such as `world-chain-el-0:9001`, so the existing
dashboard variables can be selected manually. If GitHub is unreachable while starting the devnet, the
upstream Reth dashboard is skipped with a warning and the rest of the devnet still starts.

## Remaining Gaps

- Full fault-proof game execution still needs generated Cannon prestates matching the local OP deployment. `op-challenger` is opt-in until that exists.
- HA failover is wired through op-conductor, but failover behavior is not yet covered by an automated test.
- Stable ports are available through `--stable-ports`; tests should continue to use dynamic ports.
- `FeeEscrow` and `FeeRecipient` are not deployed by the native devnet.
- Rundler and the multi-client EL/CL matrix are deferred until a specific test or workflow needs them.

## Legacy Kurtosis Devnet

The old Kurtosis path remains under `pkg/devnet` and can still be run explicitly:

```bash
just devnet-up
just ./pkg/devnet/devnet-down
```

That path builds and runs the Docker image and preserves the older service wiring. Keep using the
native `just devnet up` path for current local HA sequencing work.
