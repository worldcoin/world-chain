## Abstract

A **WorldID Gas Accounting** system enables per-credential gas subsidies for verified humans on World Chain. Each [World ID](https://worldcoin.org/world-id) gets a per-period nullifier derived from a World ID 4.0 [Uniqueness Proof](https://datatracker.ietf.org/doc/rfc9497/); budget accrues in Gwei as credential proofs are submitted and is consumed at transaction execution by addresses authorized under that nullifier. Budgets are governance-configurable per `issuerSchemaId`, authorized addresses may be added or removed throughout the period, and nullifiers expire at period boundaries.

Authorized addresses may be legacy EOAs, smart contract accounts, or [WIP-1001](./wip-1001.md) *World ID Key Rings* — the subsidy system is orthogonal to the account type it funds. A Key Ring's address is a standard 20-byte account and MAY be authorized in the Gas Accounting authorization map like any other address.

## Motivation

World ID holders are currently already being subsidized for their gas usage on World Chain. Natively issued gas subsidies allow for a more fluid fee market, and enshrine gas allowances for World ID holders at the protocol (or builder) level rather than relying on off-chain paymasters.

Per-credential budgeting lets subsidy weight reflect credential strength — a Proof-of-Human (Orb) credential carries more weight than a Phone credential, and governance can tune each independently via `issuerSchemaId`.

### Deployment Model

The Gas Accounting component can be deployed as either:

- **A precompile**, enabling the protocol to natively read gas accounting state during transaction validation and execution. This is required if gas subsidies are enforced at the protocol level (e.g., the EVM itself checks and deducts budget).
- **A standard contract** (e.g., a predeploy), where the builder reads contract state via `eth_call` during block construction to determine subsidy eligibility and track budget consumption. This avoids protocol-level changes and allows iterating on subsidy logic without hard forks.

The interface is identical in both cases. The choice of deployment model determines where enforcement happens (protocol vs. builder) but does not affect the accounting logic itself.

### Gas Accounting Interface

Manages all gas accounting state:

- **Authorization map:** maps authorized account addresses to nullifiers. One account can be mapped to multiple nullifiers.
- **Gas budget map:** maps nullifiers to remaining budget in Gwei. Budget accumulates as credential Uniqueness Proofs are submitted.
- **Credential budget configuration:** maps `issuerSchemaId` to claimable budget amount (governance-configurable per credential type).
- **Claim tracking:** tracks which `issuerSchemaId` have been claimed per nullifier per period to prevent double-claims.

Exposes methods for:

- Proof submission (verify Uniqueness Proof, claim budget and/or update authorized addresses)
- Budget lookup (remaining budget for a nullifier)
- Budget consumption (called by the protocol or builder during transaction execution)

```solidity
interface IGasAccounting {
    /// @notice Submit a World ID 4.0 Uniqueness Proof to claim budget and/or update authorized addresses.
    ///         If the credential's issuerSchemaId has not been claimed for this nullifier
    ///         in the current period, the corresponding budget is added.
    ///         If addAddresses / removeAddresses are non-empty, the authorization map is updated.
    function submitProof(
        uint256 nullifier,
        uint256 issuerSchemaId,
        bytes calldata proof,
        bytes calldata signal,
        address[] calldata addAddresses,
        address[] calldata removeAddresses
    ) external;

    /// @notice Get remaining gas budget for a nullifier in the current period.
    function getBudget(uint256 nullifier) external view returns (uint256 remainingGwei);

    /// @notice Check whether an address is authorized under a given nullifier.
    function isAuthorized(address account, uint256 nullifier) external view returns (bool);

    /// @notice Get all nullifiers associated with an address.
    function getNullifiers(address account) external view returns (uint256[] memory);

    /// @notice Check whether a credential has been claimed for a nullifier this period.
    function isClaimed(uint256 nullifier, uint256 issuerSchemaId) external view returns (bool);

    /// @notice Consume budget for a nullifier. Called during tx execution (by protocol or builder).
    function consumeBudget(uint256 nullifier, uint256 gasUsed, uint256 baseFee) external;

    /// @notice Set the claimable budget amount for a credential type. Governance only.
    function setCredentialBudget(uint256 issuerSchemaId, uint256 budgetGwei) external;
}
```

#### Authorization Map

Maps authorized account addresses to nullifiers. One account can be mapped to multiple nullifiers. Authorized accounts can be added and removed throughout the entire period via `submitProof`. Authorized addresses MAY be EOAs, smart contract accounts, or [WIP-1001](./wip-1001.md) *World ID Key Rings*.

#### Gas Budget Map

Maps nullifiers to remaining budget in Gwei. Budget accumulates as new credential Uniqueness Proofs are submitted. Tracks which `issuerSchemaId` have been claimed per nullifier per period to prevent double-claims.

### Periods and Nullifiers

The gas budget is bound to a per-period nullifier derived from a World ID 4.0 Uniqueness Proof:

- Relying party is World Chain (e.g., `rpId = 480`)
- Action is `"period_proof" || periodNumber`
- Signal contains the action (add or remove) and the list of authorized addresses

Every Uniqueness Proof natively carries an `issuerSchemaId` (mandatory public input in the circuit). The nullifier is credential-independent: the same action produces the same nullifier regardless of which credential is used. This means all credential proofs for a given World ID in a given period map to the same nullifier.

A period could be e.g. one month. It likely makes sense to divide the period into multiple slots with sub-budgets for each slot to prevent traffic spikes (e.g., when all users claim their monthly budget at once after a token launch).

Per-proof behavior of the component:

1. If the `issuerSchemaId` has not been claimed for this nullifier in the current period, claim that credential's budget.
2. If the signal contains authorized addresses, update the authorization map (add or remove).
3. Both can happen in the same proof, or either independently.

A proof with a previously-claimed credential can be resubmitted to update authorized accounts without re-claiming budget. A proof with a new credential and empty authorized accounts list can be submitted just to claim additional budget.

#### Example: Authorization and Budget Claim Flow

Assume governance has configured the following credential budgets:

| Credential           | `issuerSchemaId` | Budget      |
| -------------------- | ---------------- | ----------- |
| Proof-of-Human (Orb) | `0x01`           | 50,000 Gwei |
| Phone                | `0x02`           | 10,000 Gwei |

A user holds a World ID with both credentials. At the start of period 7, they want to authorize two addresses (`0xAlice`, `0xBob`) and claim their full budget.

**Step 1: First proof — authorize addresses and claim PoH budget.**

The user generates a Uniqueness Proof with:

- Action: `"period_proof" || 7`
- Credential: Proof-of-Human (`issuerSchemaId = 0x01`)
- Signal: `{action: add, addresses: [0xAlice, 0xBob]}`

They call `submitProof(nullifier, 0x01, proof, signal, [0xAlice, 0xBob], [])`. The component:

1. Verifies the proof.
2. `0x01` has not been claimed for this nullifier in period 7 → adds 50,000 Gwei to the budget.
3. Adds `0xAlice → nullifier` and `0xBob → nullifier` to the authorization map.

State after step 1: budget = 50,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 2: Second proof — claim phone credential budget.**

The user generates another proof with the same action (same period, so same nullifier) but a different credential:

- Credential: Phone (`issuerSchemaId = 0x02`)
- Signal: empty (no address changes)

They call `submitProof(nullifier, 0x02, proof, signal, [], [])`. The component:

1. Verifies the proof.
2. `0x02` has not been claimed for this nullifier in period 7 → adds 10,000 Gwei to the budget.
3. No address changes.

State after step 2: budget = 60,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 3 (later): Revoke an address.**

The user wants to remove `0xBob`. They resubmit a proof with any credential (e.g., PoH again):

- Signal: `{action: remove, addresses: [0xBob]}`

They call `submitProof(nullifier, 0x01, proof, signal, [], [0xBob])`. The component:

1. Verifies the proof.
2. `0x01` was already claimed → no budget change.
3. Removes `0xBob → nullifier` from the authorization map. `0xBob` can no longer consume this budget.

State after step 3: budget = 60,000 Gwei (minus any consumed so far), authorized = `[0xAlice]`.

### Budget Refresh

A nullifier with its budget and authorized accounts list is active for exactly one period. At the end of the period, all nullifiers expire.

To prevent congestion from all accounts submitting refresh proofs at the start of a new period, refresh proofs for the next period can be submitted during the current period.

Note that nullifiers of the same World ID across different periods cannot be linked. If a World ID authorizes independent addresses in each period, it remains fully anonymous.

### Gas Accounting Flow

For an incoming transaction:

1. The protocol (or builder) looks up whether the sender address has an associated nullifier in the authorization map.
2. If the address maps to more than one nullifier, a deterministic rule selects which nullifier's budget to use. A [WIP-1001](./wip-1001.md) `0x1D` transaction MAY extend its envelope with an OPTIONAL `gas_nullifier` field; if present and the `keyring` is authorized under that nullifier, the declared budget is consumed.
3. The budget is updated based on `gasUsed * baseFee` of the transaction.

A `0x1D` transaction MAY also explicitly opt out of budget consumption via the same extension path.

## Rationale

**Gas accounting deployment flexibility.** The component is designed with an identical interface whether deployed as a precompile or a standard contract (e.g., predeploy). As a contract with builder-level enforcement, it avoids protocol changes and allows faster iteration. As a precompile, it enables in-protocol enforcement. Starting with a contract is a pragmatic first step; migration to a precompile is straightforward if needed.

**Nullifier-based budget tracking.** Using World ID nullifiers for budget tracking achieves sybil resistance (one budget per World ID per period) while preserving anonymity (nullifiers cannot be linked across periods). The credential-independent nullifier means a World ID only needs one proof per period regardless of how many credentials it claims budget from.

**Per-credential budgets.** Different credentials represent different levels of verification. Proof-of-Human from an Orb carries more weight than a phone credential. Allowing governance to configure budget amounts per `issuerSchemaId` enables fine-grained subsidy policy.

**Periodic refresh with early submission.** Fixed periods with expiring nullifiers provide a clean budget lifecycle. Allowing early refresh prevents a thundering herd at period boundaries.

**Orthogonality to Key Rings.** [WIP-1001](./wip-1001.md) *World ID Key Rings* are authorized here like any other address — no special coupling is required. Legacy EOAs can opt in to gas subsidies without migrating account types, and Key Rings can sign `0x1D` transactions that either do or do not consume subsidized gas.

## Backwards Compatibility

This WIP introduces a new protocol/builder-level accounting system. It does not modify the semantics of any existing transaction type. Any account type — legacy EOA, smart contract, or [WIP-1001](./wip-1001.md) Key Ring — may be authorized under a nullifier and thereby consume subsidized gas. Whether [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) transactions from authorized accounts can consume gas budgets is an open question (see optional requirements).

## Security Considerations

**Anonymity.** Nullifiers are unlinkable across periods by construction. A World ID that authorizes different addresses each period cannot be tracked. However, if the same addresses are reauthorized across periods, on-chain observers can infer continuity.

**Cross-period linkability via address reuse.** If the same EOA is authorized in consecutive periods, an observer sees the address hit `submitProof` twice; while the nullifiers differ and are unlinkable by ZK, the address itself is a deanonymization vector. Clients SHOULD rotate authorized addresses across periods where possible — e.g., via [WIP-1001](./wip-1001.md) Key Rings at distinct nonces.

**Refresh congestion.** Even with early refresh support, a significant fraction of users may refresh near period boundaries. Sub-period slots with proportional sub-budgets can mitigate traffic spikes.

**Gas accounting trust boundary.** The `consumeBudget` method must be access-controlled. As a precompile, it must only be callable by the protocol during transaction execution. As a contract, it must be restricted to the builder's designated caller (e.g., via `onlyOwner` or a builder-specific access control mechanism).

**Uniqueness Proof replay.** A per-period nullifier is not marked spent as one-shot; the system tolerates re-submission of the same proof to update authorized addresses without re-claiming budget. A griefer observing a proof in the mempool MAY front-run the original submitter. The resulting state is identical, but ordering and gas cost shift. Implementations MAY bind the proof to `msg.sender` if stronger guarantees are required.

**Governance capture of budget parameters.** `setCredentialBudget` is governance-controlled. Misuse (e.g. setting an extravagant budget for a cheap credential) could drain subsidy pools. Governance timelocks or caps SHOULD apply.
