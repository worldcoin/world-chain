## Abstract

A **WorldID Gas Accounting** system enables per-credential gas subsidies for verified humans on World Chain. A [World ID](https://worldcoin.org/world-id) claims budget into a subsidy `sessionId` using World ID 4.0 [Uniqueness Proofs](https://datatracker.ietf.org/doc/rfc9497/); budget then accrues in Gwei as additional credential proofs are submitted and is consumed at transaction execution by addresses authorized under that `sessionId`. Budgets are governance-configurable per `issuerSchemaId`, authorized addresses may be added or removed throughout the period, and subsidy sessions expire at period boundaries.

Authorized addresses may be legacy EOAs, smart contract accounts, or [WIP-1001](./wip-1001.md) *World ID Key Rings* — the subsidy system is orthogonal to the account type it funds. A Key Ring's address is a standard 20-byte account and MAY be authorized in the Gas Accounting authorization map like any other address.

## Motivation

World ID holders are currently already being subsidized for their gas usage on World Chain. Natively issued gas subsidies allow for a more fluid fee market, and enshrine gas allowances for World ID holders at the protocol (or builder) level rather than relying on off-chain paymasters.

Per-credential budgeting lets subsidy weight reflect credential strength — a Proof-of-Human (Orb) credential carries more weight than a Phone credential, and governance can tune each independently via `issuerSchemaId`.

### Deployment Model

The Gas Accounting component can be deployed as either:

- **A precompile**, enabling the protocol to natively read gas accounting state during transaction validation and execution. This is required if gas subsidies are enforced at the protocol level (e.g., the EVM itself checks and deducts budget).
- **A standard contract** (e.g., a predeploy), where the builder reads contract state via `eth_call` during block construction to determine subsidy eligibility and track budget consumption. This avoids protocol-level changes and allows iterating on subsidy logic without hard forks.

The interface is identical in both cases. The choice of deployment model determines where enforcement happens (protocol vs. builder) but does not affect the accounting logic itself.

### Gas Accounting Interface

Manages all gas accounting state:

- **Authorization map:** maps authorized account addresses to `sessionId` values. One account can be mapped to multiple subsidy sessions.
- **Gas budget map:** maps `sessionId` values to remaining budget in Gwei. Budget accumulates as credential proofs are submitted under that subsidy session.
- **Credential budget configuration:** maps `issuerSchemaId` to claimable budget amount (governance-configurable per credential type).
- **Claim tracking:** tracks which `(issuerSchemaId, nullifier)` pairs have been claimed.

Exposes methods for:

- Subsidy claims (verify a per-period Uniqueness Proof, optionally open a `sessionId`, and credit budget)
- Address updates (verify Session Proofs and update authorized addresses)
- Budget lookup (remaining budget for a `sessionId` or authorized address)
- Budget consumption (called by the protocol or builder during transaction execution)

```solidity
interface IGasAccounting {
    /// @notice Claims the subsidy for a specific credential for the given sessionId.
    ///         MUST revert if `(issuerSchemaId, nullifier)` has already been used to claim for this period.
    ///         If the credential's issuerSchemaId has not been claimed for the current period, the corresponding budget is added to the sessionId.
    ///         The sessionId is not cryptographically tied to the World ID that generates the proof.
    function claimSubsidy(
        uint256 sessionId,
        uint256 issuerSchemaId,
        uint256 nullifier,
        bytes calldata proof,
        bytes calldata signal
    ) external returns (uint256 createdSessionId);

    /// @notice Updates the subsidized addresses corresponding to a particular session.
    function updateAddresses(
        uint256 sessionId,
        uint256 issuerSchemaId,
        uint256 sessionNullifier,
        bytes calldata proof,
        bytes calldata signal,
        address[] calldata addAddresses,
        address[] calldata removeAddresses
    ) external;

    /// @notice Get remaining gas budget for a session in the current period.
    function getBudget(uint256 sessionId) external view returns (uint256 remainingGwei);

    /// @notice Get remaining gas budget available to an address in the current period.
    ///         If the address maps to multiple sessions, the same deterministic selection rule
    ///         used during gas consumption applies here.
    function getBudget(address account) external view returns (uint256 remainingGwei);

    /// @notice Check whether an address is authorized under a given subsidy session.
    function isAuthorized(address account, uint256 sessionId) external view returns (bool);

    /// @notice Get all subsidy sessions associated with an address.
    function getSessionIds(address account) external view returns (uint256[] memory);

    /// @notice Check whether a credential has been claimed for a session this period.
    function isClaimed(uint256 sessionId, uint256 issuerSchemaId) external view returns (bool);

    /// @notice Consume budget for a subsidy session. Called during tx execution (by protocol or builder).
    function consumeBudget(uint256 sessionId, uint256 gasUsed, uint256 baseFee) external;

    /// @notice Check whether a per-period `(issuerSchemaId, nullifier)` pair has already been used.
    function isNullifierUsed(uint256 issuerSchemaId, uint256 nullifier) external view returns (bool);

    /// @notice Set the claimable budget amount for a credential type. Governance only.
    function setCredentialBudget(uint256 issuerSchemaId, uint256 budgetGwei) external;
}
```

#### Authorization Map

Maps authorized account addresses to subsidy `sessionId` values. One account can be mapped to multiple sessions. Authorized accounts can be added and removed throughout the entire period via `updateAddresses`. Authorized addresses MAY be EOAs, smart contract accounts, or [WIP-1001](./wip-1001.md) *World ID Key Rings*.

#### Gas Budget Map

Maps subsidy `sessionId` values to remaining budget in Gwei. Budget accumulates as new credential proofs are submitted. Tracks which `issuerSchemaId` have been claimed per `sessionId` per period to prevent double-claims. A helper lookup by authorized address MAY resolve the budget through the same deterministic session-selection rule used during transaction execution.

#### Nullifier Map

Maps per-period `(issuerSchemaId, nullifier)` pairs to a used bit (or associated `sessionId`). This map is only for replay prevention; subsidy budget and authorization state are keyed by `sessionId`, not by nullifier.

### Periods, Nullifiers, and Sessions

Subsidy claims are bound to a per-period nullifier derived from a World ID 4.0 Uniqueness Proof:

- Relying party is World Chain (e.g., `rpId = 480`)
- Action is `"period_proof" || periodNumber`
- Signal contains the claim context for the relying party

Every Uniqueness Proof natively carries an `issuerSchemaId` (mandatory public input in the circuit). The nullifier is credential-independent: the same period action produces the same nullifier regardless of which credential is used. Reuse prevention therefore tracks the pair `(issuerSchemaId, nullifier)`, allowing distinct credential claims in the same period while still rejecting replay of the same credential claim.

On successful `claimSubsidy`, the caller supplies a `sessionId` for `WORLD_CHAIN_RP_ID`. Budget, authorization state, and claim tracking for that period are then keyed by `sessionId` rather than by nullifier. Clients that want a fresh on-chain pseudonym MAY create a new session by choosing a fresh `sessionId` instead of reusing an old one.

##### Session Proof Primer

A Session Proof proves continuity with the same stored `sessionId` without reusing the per-period claim action. The `sessionId` is the long-lived identifier retained for the subsidy session during the period. The `sessionNullifier` is fresh per proof, passed only to `verifySession`, and discarded after verification.

A period could be e.g. one month. It likely makes sense to divide the period into multiple slots with sub-budgets for each slot to prevent traffic spikes (e.g., when all users claim their monthly budget at once after a token launch).

Per-proof behavior of the component:

For `claimSubsidy`:

1. Verify the Uniqueness Proof for `"period_proof" || periodNumber`.
2. Require that `(issuerSchemaId, nullifier)` has not been used before in the current period.
3. Mark `(issuerSchemaId, nullifier)` as used and associate the supplied `sessionId` with the subsidy session if it is newly created.
4. If the `issuerSchemaId` has not been claimed in the current period, claim that credential's budget.

For `updateAddresses`:

1. Verify a Session Proof against the stored `sessionId`.
2. Use the supplied `issuerSchemaId` as part of Session Proof verification.
3. If the signal contains authorized addresses, update the authorization map (add or remove).

A previously-claimed credential in the same period MUST NOT be reused in `claimSubsidy`, because the corresponding `(issuerSchemaId, nullifier)` pair is already consumed. `updateAddresses` changes authorization only; it does not mint additional budget.

#### Example: Authorization and Budget Claim Flow

Assume governance has configured the following credential budgets:

| Credential           | `issuerSchemaId` | Budget      |
| -------------------- | ---------------- | ----------- |
| Proof-of-Human (Orb) | `0x01`           | 50,000 Gwei |
| Phone                | `0x02`           | 10,000 Gwei |

A user holds a World ID with both credentials. At the start of period 7, they want to claim their full budget into `sessionId7` and authorize two addresses (`0xAlice`, `0xBob`).

**Step 1: Claim PoH budget into a fresh subsidy session.**

The user generates a Uniqueness Proof with:

- Action: `"period_proof" || 7`
- Credential: Proof-of-Human (`issuerSchemaId = 0x01`)
- Signal: `{action: claim}`
- Fresh `sessionId = sessionId7`

They call `claimSubsidy(sessionId7, 0x01, nullifier, proof, signal)`. The component:

1. Verifies the proof.
2. `(0x01, nullifier)` has not yet been used in period 7 → marks it used and associates it with `sessionId7`.
3. `0x01` has not been claimed for `sessionId7` in period 7 → adds 50,000 Gwei to the budget.

State after step 1: `sessionId = sessionId7`, budget = 50,000 Gwei, authorized = `[]`.

**Step 2: Session Proof — authorize addresses.**

The user generates a Session Proof against `sessionId7` with a fresh `sessionNullifier2`:

- Credential: Proof-of-Human (`issuerSchemaId = 0x01`)
- Signal: `{action: add, addresses: [0xAlice, 0xBob]}`

They call `updateAddresses(sessionId7, 0x01, sessionNullifier2, proof, signal, [0xAlice, 0xBob], [])`. The component:

1. Verifies the proof.
2. Uses `0x01` as part of Session Proof verification.
3. Adds `0xAlice → sessionId7` and `0xBob → sessionId7` to the authorization map.

State after step 2: budget = 50,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 3: Claim phone credential budget into the same session.**

The user now claims the Phone credential into the same `sessionId7`:

- Credential: Phone (`issuerSchemaId = 0x02`)
- Signal: `{action: claim}`

They call `claimSubsidy(sessionId7, 0x02, nullifier, proof, signal)`. The component:

1. Verifies the proof.
2. `(0x02, nullifier)` has not yet been used in period 7, even though the underlying `nullifier` matches step 1.
3. `0x02` has not been claimed for `sessionId7` in period 7 → adds 10,000 Gwei to the budget.

State after step 3: budget = 60,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 4 (later): Session Proof — revoke an address.**

The user wants to remove `0xBob`. They submit another Session Proof under `sessionId7` with a fresh `sessionNullifier3`:

- Credential: Proof-of-Human (`issuerSchemaId = 0x01`)
- Signal: `{action: remove, addresses: [0xBob]}`

They call `updateAddresses(sessionId7, 0x01, sessionNullifier3, proof, signal, [], [0xBob])`. The component:

1. Verifies the proof.
2. Uses `0x01` as part of Session Proof verification.
3. Removes `0xBob → sessionId7` from the authorization map. `0xBob` can no longer consume this budget.

State after step 4: budget = 60,000 Gwei (minus any consumed so far), authorized = `[0xAlice]`.

### Budget Refresh

A subsidy associated with a `sessionId` and authorized accounts list is active for exactly one period. At the end of the period, all subsidies expire.

To prevent congestion from all accounts submitting refresh proofs at the start of a new period, the next period's initial `claimSubsidy` can be submitted during the current period.

Note that the claim nullifiers of the same World ID across different periods cannot be linked. Subsidy `sessionId` values may be used across periods for convenience. More generally, a user that wants a new on-chain pseudonym can simply create a new session instead of reusing an old `sessionId`. If a World ID authorizes independent addresses and fresh sessions in each period, it remains fully anonymous.

### Gas Accounting Flow

For an incoming transaction:

1. The protocol (or builder) looks up whether the sender address has an associated subsidy `sessionId` in the authorization map.
2. If the address maps to more than one `sessionId`, a deterministic rule selects which session's budget to use. A [WIP-1001](./wip-1001.md) `0x1D` transaction MAY extend its envelope with an OPTIONAL `gas_session_id` field; if present and the `keyring` is authorized under that `sessionId`, the declared budget is consumed.
3. The budget is updated based on `gasUsed * baseFee` of the transaction.

`getBudget(address)` SHOULD apply the same deterministic session-selection rule so off-chain callers observe the same effective budget that transaction execution would consume.

A `0x1D` transaction MAY also explicitly opt out of budget consumption via the same extension path.

## Rationale

**Gas accounting deployment flexibility.** The component is designed with an identical interface whether deployed as a precompile or a standard contract (e.g., predeploy). As a contract with builder-level enforcement, it avoids protocol changes and allows faster iteration. As a precompile, it enables in-protocol enforcement. Starting with a contract is a pragmatic first step; migration to a precompile is straightforward if needed.

**SessionId-based budget tracking with `(issuerSchemaId, nullifier)` replay protection.** Budget and authorization state live under `sessionId`, while proof reuse is prevented by tracking `(issuerSchemaId, nullifier)` rather than the nullifier alone. This allows the same World ID to claim distinct credentials in the same period even though the underlying nullifier is credential-independent, while still rejecting replay of the same credential claim.

**Per-credential budgets.** Different credentials represent different levels of verification. Proof-of-Human from an Orb carries more weight than a phone credential. Allowing governance to configure budget amounts per `issuerSchemaId` enables fine-grained subsidy policy.

**Periodic refresh with early submission.** Fixed periods with expiring subsidies provide a clean budget lifecycle. Allowing early refresh prevents a thundering herd at period boundaries.

**Orthogonality to Key Rings.** [WIP-1001](./wip-1001.md) *World ID Key Rings* are authorized here like any other address — no special coupling is required. Legacy EOAs can opt in to gas subsidies without migrating account types, and Key Rings can sign `0x1D` transactions that either do or do not consume subsidized gas.

## Backwards Compatibility

This WIP introduces a new protocol/builder-level accounting system. It does not modify the semantics of any existing transaction type. Any account type — legacy EOA, smart contract, or [WIP-1001](./wip-1001.md) Key Ring — may be authorized under a subsidy `sessionId` and thereby consume subsidized gas. Whether [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) transactions from authorized accounts can consume gas budgets is an open question (see optional requirements).

## Security Considerations

**Anonymity.** Claim nullifiers are unlinkable across periods by construction. A World ID that uses fresh subsidy `sessionId` values and authorizes different addresses each period cannot be tracked. If a user wants a new on-chain pseudonym, they can simply create a new session instead of reusing an old `sessionId`. However, if the same addresses or the same `sessionId` are reused across periods, on-chain observers can infer continuity.

**Cross-period linkability via address or session reuse.** If the same EOA is authorized in consecutive periods, an observer sees the address hit the subsidy system twice; while the claim nullifiers differ and are unlinkable by ZK, the address itself is a deanonymization vector. Reusing the same `sessionId` across periods would similarly create a stable identifier. Clients may rotate authorized addresses and use fresh subsidy sessions across periods where possible — e.g., via [WIP-1001](./wip-1001.md) Key Rings at distinct nonces.

**Refresh congestion.** Even with early refresh support, a significant fraction of users may refresh near period boundaries. Sub-period slots with proportional sub-budgets can mitigate traffic spikes.

**Gas accounting trust boundary.** The `consumeBudget` method must be access-controlled. As a precompile, it must only be callable by the protocol during transaction execution. As a contract, it must be restricted to the builder's designated caller (e.g., via `onlyOwner` or a builder-specific access control mechanism).

**Claim nullifier reuse and Session Proof replay.** Claim nullifiers are tracked as `(issuerSchemaId, nullifier)` pairs and MUST NOT be reused within the same period. Session Proofs may still be replayed in-flight by a griefer; signal binding prevents substituting a different update, so the resulting state is identical, but ordering and gas cost shift. Implementations MAY bind proofs to `msg.sender` if stronger guarantees are required.

**Governance capture of budget parameters.** `setCredentialBudget` is governance-controlled. Misuse (e.g. setting an extravagant budget for a cheap credential) could drain subsidy pools. Governance timelocks or caps SHOULD apply.
