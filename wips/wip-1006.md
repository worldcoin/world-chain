## Abstract

This WIP specifies an optional [reth Execution Extension][exex] (ExEx),
`world-chain-relayer`, that runs inside the World Chain node and automates the
L2-to-L1 withdrawal lifecycle defined by the [OP Stack withdrawals
specification][op-withdrawals].

The ExEx has two cooperating roles:

1. **Cacher** — observes committed L2 blocks through the ExEx notification
   stream, extracts `MessagePassed` events emitted by the `L2ToL1MessagePasser`
   predeploy, and persists each withdrawal together with the metadata needed to
   later construct its inclusion proof. The cache is reorg-aware.
2. **Relayer** — a driver loop that, for each cached withdrawal, waits for an L2
   output covering the withdrawal's block to become available on L1 (a resolved
   dispute game), builds the `OutputRootProof` and the Merkle-Patricia storage
   proof, submits `proveWithdrawalTransaction`, and after the proof-maturity and
   dispute-game air-gap delays submits `finalizeWithdrawalTransaction`.

Both roles are disabled by default and enabled via CLI flags under the
`--relayer.*` namespace, mirroring the existing OP Proposer ExEx
([`crates/exex`][proposer-crate]). The relayer is the natural counterpart of the
proposer: the proposer publishes the output proposals (dispute games) — under
the fault-proof model of [WIP-1005] — that the relayer consumes when proving
withdrawals.

## Motivation

On the OP Stack, a withdrawal is a three-step, asynchronous process spread
across two chains and gated by multi-day delays (see
[withdrawal flow][op-withdrawal-flow]):

1. `initiateWithdrawal` on L2 (`L2ToL1MessagePasser`),
2. `proveWithdrawalTransaction` on L1 once an output root covering the
   withdrawal's block has been proposed, and
3. `finalizeWithdrawalTransaction` on L1 once the proof has matured and the
   backing dispute game has resolved and cleared the air-gap.

Constructing the proof in step 2 requires an `eth_getProof` against the
`L2ToL1MessagePasser` account at the exact block that backs the chosen output
root, plus the four-field `OutputRootProof` preimage for that block. Performed
out-of-band, this requires an archive endpoint, an indexer that maps
withdrawals to output proposals, and a long-lived process to shepherd each
withdrawal across the maturity windows.

A node already holds the canonical L2 state and receives every committed block
via the ExEx stream, so it is the cheapest possible place to index withdrawals
and source proofs. Co-locating the relayer with the node removes the external
archive/indexer dependency and lets operators run a self-contained automatic
withdrawal service with a single flag.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",

This document mirrors the OP Stack withdrawal contracts and proof construction.
Contract names, storage layout, and delay semantics are pinned to the upstream

### Cacher

The cacher MUST consume the ExEx notification stream and react to each variant:

- On `ChainCommitted { new }` and the `new` side of `ChainReorged { old, new }`,
  the cacher MUST scan every receipt in the committed range for logs emitted by
  the `L2ToL1MessagePasser` predeploy at
  `0x4200000000000000000000000000000000000016` whose topic0 is the
  `MessagePassed` event selector. Each matching log MUST be decoded into a
  withdrawal record.
- On `ChainReverted { old }` and the `old` side of `ChainReorged`, the cacher
  MUST delete every cached withdrawal whose origin block falls within the
  reverted range, unless that withdrawal has already been proven or finalized on
  L1 (such records MUST be retained for auditing but marked as orphaned).
- After processing each notification the ExEx MUST forward
  `ExExEvent::FinishedHeight` for the committed tip so reth may prune up to that
  height. The cacher MUST NOT acknowledge a height until the corresponding
  withdrawals are durably persisted.

Each `MessagePassed` event carries the fields of the withdrawal transaction
(`nonce`, `sender`, `target`, `value`, `gasLimit`, `data`) and the precomputed
`withdrawalHash`. The cacher MUST persist a record keyed by `withdrawalHash`
containing at minimum:

| Field              | Source                                                        |
| ------------------ | ------------------------------------------------------------- |
| `withdrawal_hash`  | `MessagePassed.withdrawalHash`                                |
| `tx`               | decoded `WithdrawalTransaction` fields                        |
| `l2_block_number`  | block that emitted the event                                  |
| `l2_block_hash`    | block hash (for reorg detection)                              |
| `message_slot`     | `keccak256(abi.encode(withdrawalHash, uint256(0)))` — the `sentMessages` mapping slot |
| `status`           | `Cached → Proven → Finalized` (or `Orphaned`)                 |

The `withdrawalHash` MUST equal `keccak256(abi.encode(tx))`; the cacher SHOULD
recompute it from the decoded fields and reject any record where the recomputed
hash does not match the event-supplied hash.

State MUST be persisted to a dedicated MDBX store under the relayer datadir, so
that progress survives restarts. The store MUST be idempotent: re-observing an
already-cached `withdrawalHash` MUST NOT create a duplicate.

### Proof construction

For a cached withdrawal at L2 block `B`, the relayer MUST construct the proof
artifacts required by `proveWithdrawalTransaction` exactly as defined by the
[withdrawals spec][op-withdrawals]:

1. **`OutputRootProof`** for block `B`, the four-field preimage of the output
   root:
   - `version` (`bytes32(0)` for V0),
   - `stateRoot` — the L2 state root at `B`,
   - `messagePasserStorageRoot` — the storage root of the `L2ToL1MessagePasser`
     account at `B`,
   - `latestBlockhash` — the hash of `B`.

   The output root is `keccak256(abi.encode(version, stateRoot,
   messagePasserStorageRoot, latestBlockhash))` and MUST equal the root the
   chosen dispute game commits to.
2. **`withdrawalProof`** — the RLP-encoded Merkle-Patricia inclusion proof for
   `message_slot` in the `L2ToL1MessagePasser` account's storage trie at `B`,
   obtained from the node's local state. The relayer SHOULD source this directly
   from the node's state provider (the same path `eth_getProof` uses) rather
   than over RPC, so no archive endpoint is required.

The relayer MUST select a dispute game whose committed L2 block number is
`>= B`, and pass that game's index as `_disputeGameIndex`. The
`OutputRootProof` MUST be built for the precise block the selected game commits
to (not for `B` itself when they differ); the storage proof MUST be taken at
that same block.

### Relayer driver

The driver MUST run a poll loop (interval `--relayer.poll-interval`) that
advances each cached withdrawal through its lifecycle:

1. **Awaiting output.** A `Cached` withdrawal is eligible to prove once a
   dispute game of the respected game type committing to a block `>= B` exists
   in the `DisputeGameFactory`. Until then the withdrawal MUST remain `Cached`.
2. **Prove.** When eligible, the relayer MUST build the proof artifacts
   (above), then submit `proveWithdrawalTransaction(tx, disputeGameIndex,
   outputRootProof, withdrawalProof)` to the `OptimismPortal2` from the
   configured L1 signer. On inclusion, the record transitions to `Proven` and
   records the prove timestamp and the game index used.
   - The relayer MUST handle the proof being invalidated (e.g. the backing game
     resolves `CHALLENGER_WINS` or is blacklisted) by re-proving against a
     different valid game, as permitted by `OptimismPortal2`.
3. **Finalize.** A `Proven` withdrawal becomes finalizable only when **all** of
   the following hold, per `OptimismPortal2.checkWithdrawal`:
   - the proof has matured: `now >= provenAt + PROOF_MATURITY_DELAY_SECONDS`;
   - the backing dispute game has resolved `DEFENDER_WINS`;
   - the air-gap has elapsed: `now >= gameResolvedAt +
     DISPUTE_GAME_FINALITY_DELAY_SECONDS`;
   - the game is not blacklisted and its type is the currently respected game
     type.

   When satisfied, the relayer MUST submit
   `finalizeWithdrawalTransaction(tx)`. On inclusion the record transitions to
   `Finalized`. The relayer MUST NOT submit a finalize transaction for a
   withdrawal already marked finalized on-chain (`finalizedWithdrawals`).

The relayer MUST be resilient to restarts at any point: lifecycle state is
derived from the persisted store reconciled against on-chain
`provenWithdrawals` / `finalizedWithdrawals` reads at startup.

### L1 interaction and signing

The L1 client, signer handling (private key XOR mnemonic), retry/backoff,
network timeout, and balance metrics MUST reuse the same provider stack as the
OP Proposer ExEx ([`crates/exex`][proposer-crate]). The signer secret MUST be redacted from all
`Debug` output and logs.

### CLI

The ExEx MUST be configured through the `--relayer.*` flag namespace and be
disabled unless `--relayer.enabled` is set. The node MUST install it
conditionally (`install_exex_if`), exactly as the proposer is installed in
`crates/node/src/extensions.rs`. When disabled, the ExEx MUST still drain the
notification stream and forward `FinishedHeight` so it never stalls pruning.

| Flag                                | Env                              | Default            | Meaning                                                    |
| ----------------------------------- | -------------------------------- | ------------------ | ---------------------------------------------------------- |
| `--relayer.enabled`                 | `WC_RELAYER_ENABLED`             | `false`            | Enable the withdrawal cacher + relayer ExEx.               |
| `--relayer.cache-only`              | `WC_RELAYER_CACHE_ONLY`          | `false`            | Run the cacher only; never submit L1 transactions.         |
| `--relayer.l1-eth-rpc`              | `WC_RELAYER_L1_ETH_RPC`          | _(required)_       | L1 execution RPC URL(s), comma-separated for failover.     |
| `--relayer.portal-address`          | `WC_RELAYER_PORTAL_ADDRESS`      | _(required)_       | `OptimismPortal2` address.                                 |
| `--relayer.game-factory-address`    | `WC_RELAYER_GAME_FACTORY_ADDRESS`| _(required)_       | `DisputeGameFactory` address (output source).              |
| `--relayer.game-type`               | `WC_RELAYER_GAME_TYPE`           | `0`                | Respected dispute game type to prove against.              |
| `--relayer.poll-interval`           | `WC_RELAYER_POLL_INTERVAL`       | `12s`              | Driver loop interval.                                      |
| `--relayer.private-key`             | `WC_RELAYER_PRIVATE_KEY`         | _(one of)_         | L1 signer key (XOR with mnemonic).                         |
| `--relayer.mnemonic`                | `WC_RELAYER_MNEMONIC`            | _(one of)_         | L1 signer mnemonic.                                        |
| `--relayer.hd-path`                 | `WC_RELAYER_HD_PATH`             | `m/44'/60'/0'/0/0` | HD derivation path for the mnemonic signer.                |
| `--relayer.network-timeout`         | `WC_RELAYER_NETWORK_TIMEOUT`     | `10s`              | Per-call L1 network timeout.                               |
| `--relayer.rpc-addr`                | `WC_RELAYER_RPC_ADDR`            | `127.0.0.1`        | Admin RPC bind address.                                    |
| `--relayer.rpc-port`                | `WC_RELAYER_RPC_PORT`            | `0` (disabled)     | Admin RPC bind port.                                       |
| `--relayer.datadir`                 | `WC_RELAYER_DATADIR`             | `<datadir>/wc-relayer` | MDBX store location for the withdrawal cache.          |

Configuration validation MUST require: an L1 RPC, a portal address, a game
factory address, and exactly one signer unless `--relayer.cache-only` is set
(in which case the signer flags MUST be ignored and no L1 transactions are
sent).

### Admin RPC

When `--relayer.rpc-port` is non-zero, the ExEx SHOULD expose a read-only admin
namespace returning, at minimum: the cache tip height, counts by lifecycle
status, and per-withdrawal status lookup by `withdrawalHash`. The admin RPC
MUST NOT expose signer material.

### Chain Specifics

On devnets and chains running the legacy `L2OutputOracle` instead of fault
proofs, the relayer MAY substitute the oracle as the output source and use the
legacy `proveWithdrawalTransaction` overload that takes an `_l2OutputIndex`. The
delay model collapses to the single `FINALIZATION_PERIOD_SECONDS`. The cacher
behaviour is unchanged.

## Rationale

**Why an ExEx rather than a standalone binary.** Proof construction needs
`messagePasserStorageRoot` and a storage inclusion proof at a historical block.
Inside the node these come straight from the state provider with no archive RPC
and no separate indexer. Withdrawal discovery is likewise free: every committed
block already arrives over the ExEx stream. Running out-of-process would
duplicate both.

**Why mirror the proposer.** The existing OP Proposer ExEx
([`crates/exex`][proposer-crate]) already establishes the conventions this
proposal reuses wholesale — `--namespace.enabled` gating,
`install_exex_if`, the L1 provider/signer stack, MDBX persistence keyed off the
committed tip, secret redaction, and `FinishedHeight` draining when disabled.
The relayer is the proposer's consumer (it proves against the games the proposer
creates), so sharing one design surface keeps the two coherent and minimizes
new code.

**Reorg handling.** Withdrawals are only meaningful relative to a canonical L2
block. Pruning cached-but-unproven withdrawals on revert keeps the cache
consistent with the canonical chain, while retaining already-proven records
(marked orphaned) preserves an audit trail for an operator whose proof targeted
a now-reorged block.

## Backwards Compatibility

No backward compatibility issues found. The ExEx is disabled by default, adds
no consensus-relevant behaviour, and introduces only new `--relayer.*` CLI flags
and an optional admin RPC namespace. When disabled it is a no-op that only
drains the notification stream.

## Test Cases

- **Cacher decode.** Given a block whose receipts contain a `MessagePassed` log,
  the cache MUST contain exactly one record with a `withdrawalHash` equal to
  `keccak256(abi.encode(tx))`.
- **Reorg prune.** A `ChainReverted` over the block that produced a `Cached`
  withdrawal MUST remove it; a `Proven` withdrawal over the same range MUST be
  retained and marked `Orphaned`.
- **Idempotent restart.** Re-delivering an already-processed notification range
  MUST NOT create duplicates, and a restart MUST reconcile lifecycle state from
  on-chain `provenWithdrawals` / `finalizedWithdrawals`.
- **Prove gating.** A withdrawal MUST remain `Cached` until a respected-type
  dispute game committing to a block `>= B` exists.
- **Finalize gating.** A `Proven` withdrawal MUST NOT be finalized until proof
  maturity, `DEFENDER_WINS` resolution, and the air-gap have all elapsed.
- **End-to-end.** Against the devnet OP stack, an `initiateWithdrawal` on L2 MUST
  be automatically proven and then finalized on L1 within the configured delays.

## Reference Implementation

To be implemented as a sibling of the OP Proposer ExEx (`crates/exex`), reusing
its provider, signer, and MDBX-store modules. The driver and proof-construction
modules are new; the cacher hooks into the same ExEx notification loop. CLI
wiring follows `crates/node/src/extensions.rs::install_extensions`.

## Security Considerations

- **Signer key custody.** The relayer holds a funded L1 EOA. The key MUST be
  redacted from all logs and `Debug` output (as in the proposer ExEx) and SHOULD be
  loaded from a secrets manager rather than a flag. `--relayer.cache-only` lets
  operators run discovery/proof-building with no key present.
- **Proof correctness is consensus-checked.** A malformed `OutputRootProof` or
  storage proof simply reverts on L1; the relayer cannot forge a withdrawal.
  The trust assumption is solely that the node's local state is canonical —
  the same assumption the node already makes.
- **Output-source trust.** Proving against an invalid or challenger-won game
  wastes gas and can be invalidated on L1. The relayer MUST only prove against
  the respected game type and MUST re-prove if its chosen game is invalidated.
- **Griefing / gas exhaustion.** A high withdrawal rate could drain the signer.
  The relayer SHOULD bound in-flight L1 transactions and surface balance
  metrics so operators can alert before funds are exhausted.
- **No premature finalization.** Finalization gating MUST exactly reflect
  `OptimismPortal2.checkWithdrawal`; the on-chain delays are the real security
  boundary and the relayer MUST NOT attempt to bypass them.
