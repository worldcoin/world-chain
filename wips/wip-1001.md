## Abstract

This WIP introduces native account abstraction for World Chain with three core features: (1) a **WorldID Gas Accounting** system enabling per-credential gas subsidies for verified humans via a persistent World Chain session created from an initial uniqueness proof, with subsidy budgets accruing automatically each subsidy epoch while associated credentials remain valid, (2) **World ID KeyRings** — accounts whose session-key set is administered by World ID session proofs rather than by long-lived proof nullifiers, supporting native multi-authentication (Secp256k1, P256, WebAuthn, EdDSA) without bundler infrastructure, and (3) WorldID-based recovery of keyring session keys with timelock protection against compromised authenticators. A new transaction type (`0x1D`) is introduced that is signed by any authorized session key of a keyring and optionally targets a specific session subsidy budget.

A World ID KeyRing is tangential to Gas Accounting: its sole purpose is to abstract the signature scheme authenticating an account. A keyring's address is a standard 20-byte account and MAY be authorized in the Gas Accounting authorization map like any other address.



## Motivation

### WorldID Gas Accounting

World ID holders are currently already being subsidized for their gas usage on World Chain. Natively issued gas subsidies allows for a more fluid fee market, and allows the protocol to enshrine gas allowances for World ID holders.

### Native Multi-Auth

- **Multiple authentication methods.** Support for passkeys, different signature schemes, multi-sig, and WorldID as authenticators.
- **Native support.** Users should be able to submit transactions directly without bundler infrastructure.
- **Key rotation.** Authenticators can be added and removed after account creation.

### WorldID Recovery

- **Proof-based recovery.** A user can recover their account by submitting a WorldID proof, as long as they control the WorldID's authenticators.
- **Recovery means updating authenticators.** Recovery updates the set of authenticators on the account, not a migration of funds to a new address.
- **Protection from compromised authenticators.** A compromised WorldID authenticator must not immediately lead to a drained account. A safety mechanism (e.g., timelock) is required.
- **Stable proof-based control.** A user should be able to keep controlling subsidized accounts and keyrings through a persistent World ID session, without storing a long-lived admin key on-device.

### WorldID Gas Accounting

The accounting logic is implemented in a dedicated Gas Accounting component. All persistent subsidy state is keyed by a **World Chain session**. A session is created once from an initial World ID **Uniqueness Proof** and then reused via **Session Proofs**. The uniqueness nullifier from that initial enrollment proof MUST be stored on chain to prevent duplicate enrollment under the enrollment action. Once accepted, the protocol creates a long-lived `sessionId` and keys all ongoing subsidy state by that `sessionId`.

There is a governance-configurable map from credentials (identified by `issuerSchemaId`) to a per-epoch gas subsidy amount. A credential is attached to a session by successfully verifying a World ID proof with that credential. Once attached, its subsidy is issued automatically each **subsidy epoch** until the credential expires. No fresh proof is required at every epoch boundary.

The component maintains an authorization map from authorized account addresses to session ids, a gas budget map from session ids to remaining budget, and per-session credential state tracking expiry and the last subsidy epoch that has been accrued.

The gas accounting component can be deployed as either:

- **A precompile**, enabling the protocol to natively read gas accounting state during transaction validation and execution. This is required if gas subsidies are enforced at the protocol level (e.g., the EVM itself checks and deducts budget).
- **A standard contract** (e.g., a predeploy), where the builder reads contract state via `eth_call` during block construction to determine subsidy eligibility and track budget consumption. This avoids protocol-level changes and allows iterating on subsidy logic without hard forks.

The interface is identical in both cases. The choice of deployment model determines where enforcement happens (protocol vs. builder) but does not affect the accounting logic itself.

#### Proof Inputs and Outputs

The World Chain RP uses `WORLD_CHAIN_RP_ID = 480`.

Session creation starts with a Uniqueness Proof. Ongoing mutations use Session Proofs. The relevant on-chain verifier interfaces are:

```solidity
verify(
    uint256 nullifier,
    uint256 action,
    uint64 rpId,
    uint256 nonce,
    uint256 signalHash,
    uint64 expiresAtMin,
    uint64 issuerSchemaId,
    uint256 credentialGenesisIssuedAtMin,
    uint256[5] calldata proof
)

verifySession(
    uint64 rpId,
    uint256 nonce,
    uint256 signalHash,
    uint64 expiresAtMin,
    uint64 issuerSchemaId,
    uint256 credentialGenesisIssuedAtMin,
    uint256 sessionId,
    uint256[2] calldata sessionNullifier,
    uint256[5] calldata proof
)
```

##### Initial Session Creation

An initial session is created from:

1. A Uniqueness Proof response for a chosen credential.
2. A freshly generated `SessionId` returned by the authenticator.

The authenticator-generated `SessionId` is not itself an input to `verify()`. Rather, the protocol verifies the Uniqueness Proof, records the returned uniqueness `nullifier` on chain, and then persists `sessionId = SessionId.commitment()` as the stable key for future state.

Because `world-id-protocol` disallows duplicate Uniqueness Proofs for the same RP and action, the **initial enrollment Uniqueness Proof** MUST use a fixed World Chain action such as `WORLD_CHAIN_CREATE_SESSION`. This requirement applies only to the `action` field passed to `verify(...)` during enrollment. It does **not** apply to Session Proofs, whose internal action remains randomized by `world-id-protocol`. Using a fixed enrollment action and storing the resulting uniqueness nullifier on chain makes session creation one-time for a given World ID under this WIP and prevents multiplying subsidies by re-enrolling with the same credential set.

The `CreateSession` payload bound by `signalHash` SHOULD include at least `sessionId` and the initial authorization update, so the enrollment proof is tied to the exact session being persisted on chain.

| Field                            | Type           | Role |
| -------------------------------- | -------------- | ---- |
| `sessionId`                      | off-chain `SessionId` / on-chain `uint256` commitment | Stable session identifier returned by the authenticator. The protocol stores only `SessionId.commitment()`. |
| `nullifier`                      | `uint256`      | Public output of the Uniqueness Proof. MUST be recorded on chain when `createSession` succeeds to prevent duplicate enrollment. MUST NOT key ongoing subsidy state afterwards. |
| `action`                         | `uint256`      | Fixed World Chain session-creation action. |
| `rpId`                           | `uint64`       | `WORLD_CHAIN_RP_ID`. |
| `nonce`                          | `uint256`      | Fresh RP request nonce. |
| `signalHash`                     | `uint256`      | Hash of the `CreateSession` payload being authorized. |
| `expiresAtMin`                   | `uint64`       | Public proof input constraining the minimum expiration of the credential used for enrollment. |
| `issuerSchemaId`                 | `uint64`       | Credential schema attached to the new session. |
| `credentialGenesisIssuedAtMin`   | `uint256`      | Optional lower bound on the credential's genesis issuance time. |
| `proof`                          | `uint256[5]`   | Encoded World ID zero-knowledge proof. |

##### Session Proofs

Subsequent mutations use `verifySession(...)`. Session proofs are tied to the stored `sessionId`, not to the original enrollment nullifier.

`world-id-protocol` allows multiple Session Proofs for the same `sessionId` because the internal action used to produce the session nullifier is randomized for each proof. The `sessionNullifier` is therefore verifier input only and MUST NOT be persisted as protocol state.

For Session Proof entrypoints, `signalHash` SHOULD bind the exact mutation being authorized. For example, `registerCredential` SHOULD bind a `RegisterCredential(sessionId, issuerSchemaId)` payload, while `updateAuthorizedAddresses` SHOULD bind the add/remove address lists.

| Field                            | Type           | Role |
| -------------------------------- | -------------- | ---- |
| `sessionId`                      | `uint256`      | Stored session commitment proving continuity with the enrollment session. |
| `sessionNullifier[0]`            | `uint256`      | Ephemeral public output of the Session Proof. Used only for `verifySession()`. |
| `sessionNullifier[1]`            | `uint256`      | Randomized internal action required by `world-id-protocol` for Session Proofs. |
| `rpId`                           | `uint64`       | `WORLD_CHAIN_RP_ID`. |
| `nonce`                          | `uint256`      | Fresh RP request nonce. |
| `signalHash`                     | `uint256`      | Hash of the specific mutation being authorized (e.g., address update or keyring update). |
| `expiresAtMin`                   | `uint64`       | Public proof input constraining the minimum expiration of the credential used for this proof. |
| `issuerSchemaId`                 | `uint64`       | Credential schema used for this proof. |
| `credentialGenesisIssuedAtMin`   | `uint256`      | Optional lower bound on the credential's genesis issuance time. |
| `proof`                          | `uint256[5]`   | Encoded World ID zero-knowledge proof. |

#### Gas Accounting Interface

The component manages:

- **Enrollment nullifier registry:** tracks which uniqueness nullifiers have already created a session.
- **Session registry:** maps `sessionId` to session state.
- **Authorization map:** maps authorized account addresses to session ids. One account MAY be mapped to multiple sessions.
- **Gas budget map:** maps session ids to remaining budget in Gwei.
- **Credential state:** maps `(sessionId, issuerSchemaId)` to the credential's current `expiresAtMin` and the last subsidy epoch already accrued.
- **Credential budget configuration:** maps `issuerSchemaId` to subsidy amount per subsidy epoch.

```solidity
struct UniquenessProof {
    uint256 nullifier;
    uint256 action;
    uint256 nonce;
    uint256 signalHash;
    uint64  expiresAtMin;
    uint64  issuerSchemaId;
    uint256 credentialGenesisIssuedAtMin;
    uint256[5] proof;
}

struct SessionProof {
    uint256 nonce;
    uint256 signalHash;
    uint64  expiresAtMin;
    uint64  issuerSchemaId;
    uint256 credentialGenesisIssuedAtMin;
    uint256 sessionId;
    uint256[2] sessionNullifier;
    uint256[5] proof;
}

interface IGasAccounting {
    /// @notice Create a new subsidy session from an initial Uniqueness Proof.
    ///         The proof's nullifier is recorded in the enrollment nullifier registry.
    ///         The proof's credential is attached to the new session, and
    ///         any requested addresses are authorized under that session.
    function createSession(
        UniquenessProof calldata proof,
        uint256 sessionId,
        address[] calldata addAddresses
    ) external;

    /// @notice Attach a credential to an existing session or refresh its expiry.
    ///         The session proof MUST verify against `proof.sessionId`.
    function registerCredential(SessionProof calldata proof) external;

    /// @notice Update which addresses are authorized to spend a session's subsidy.
    ///         The signalHash MUST bind the add/remove payload.
    ///         The proof MAY also refresh the expiry of the credential used for this proof.
    function updateAuthorizedAddresses(
        SessionProof calldata proof,
        address[] calldata addAddresses,
        address[] calldata removeAddresses
    ) external;

    /// @notice Get remaining gas budget for a session in the current subsidy epoch.
    function getBudget(uint256 sessionId) external view returns (uint256 remainingGwei);

    /// @notice Check whether an address is authorized under a given session.
    function isAuthorized(address account, uint256 sessionId) external view returns (bool);

    /// @notice Get all sessions associated with an address.
    function getSessions(address account) external view returns (uint256[] memory);

    /// @notice Return the tracked expiry for a credential attached to a session.
    function getCredentialExpiry(uint256 sessionId, uint256 issuerSchemaId) external view returns (uint64 expiresAtMin);

    /// @notice Consume budget for a session. Called during tx execution (by protocol or builder).
    function consumeBudget(uint256 sessionId, uint256 gasUsed, uint256 baseFee) external;

    /// @notice Set the subsidy amount per epoch for a credential type. Governance only.
    function setCredentialBudget(uint256 issuerSchemaId, uint256 budgetGweiPerEpoch) external;

    /// @notice Set the subsidy epoch length. Governance only.
    function setSubsidyEpochLength(uint256 epochLengthSeconds) external;
}
```

##### Authorization Map

Maps authorized account addresses to session ids. One account MAY be mapped to multiple sessions. Authorized accounts are added during `createSession` and can later be added or removed via `updateAuthorizedAddresses`, which requires only a fresh Session Proof bound to the requested mutation.

##### Gas Budget Map

Maps session ids to remaining budget in Gwei. A session's budget is the aggregate of all attached credentials whose subsidies have been accrued through the current subsidy epoch.

#### Subsidy Epochs

Time is divided into **subsidy epochs** of governance-configurable length.

For each attached credential `(sessionId, issuerSchemaId)`:

1. Governance defines `budgetPerEpoch[issuerSchemaId]`.
2. The component tracks `lastAccruedEpoch[sessionId][issuerSchemaId]`.
3. On `createSession`, `registerCredential`, `updateAuthorizedAddresses`, or `consumeBudget`, the component lazily settles any unaccrued subsidy epochs up to the current epoch. `getBudget` MUST return the budget as if those epochs had already been settled.
4. For each subsidy epoch whose start time is strictly before the credential's tracked `expiresAtMin`, the component adds `budgetPerEpoch[issuerSchemaId]` to the session budget.
5. Once `expiresAtMin` is reached, no subsidy is issued for subsequent subsidy epochs until a later proof refreshes or re-attaches that credential.

This means a credential that expires during subsidy epoch `N` MAY still have budget that was issued for epoch `N`, but it MUST NOT receive budget for epoch `N + 1` or later unless a new proof refreshes the credential expiry.

The subsidy epoch is a budget reset cadence, not a proof cadence. A user does **not** need to submit a new proof every epoch as long as the credential attached to the session remains valid.

#### Credential Attachment Rules

- `createSession` attaches the credential used in the enrollment Uniqueness Proof.
- `registerCredential` attaches a new credential to an existing session, or refreshes the tracked expiry for an already-attached credential.
- A single session MAY have multiple attached credentials. The session budget for any epoch is the sum of the active per-credential epoch subsidies.
- A Session Proof with a later `expiresAtMin` SHOULD refresh the stored expiry for that `(sessionId, issuerSchemaId)`.
- Refreshing a credential after an expiry gap MUST NOT backfill subsidy epochs that started after the old expiry and before the refresh.

#### Example: Session, Automatic Subsidy Issuance, and Address Updates

Assume governance has configured the following subsidy amounts:

| Credential           | `issuerSchemaId` | Subsidy / epoch |
| -------------------- | ---------------- | --------------- |
| Proof-of-Human (Orb) | `0x01`           | 50,000 Gwei     |
| Phone                | `0x02`           | 10,000 Gwei     |

A user holds both credentials and wants to subsidize two addresses, `0xAlice` and `0xBob`.

**Step 1: Create the session with a Uniqueness Proof.**

The user requests a World Chain session from the authenticator and receives:

- A Uniqueness Proof for the Orb credential.
- A new `SessionId`, of which the protocol stores only `sessionId = SessionId.commitment()`.

They call `createSession(proof, sessionId, [0xAlice, 0xBob])`. The component:

1. Verifies the Uniqueness Proof via `verify(...)`.
2. Creates session `sessionId`.
3. Attaches Orb (`0x01`) to the session with its `expiresAtMin`.
4. Accrues the current subsidy epoch for Orb if the credential is valid for that epoch.
5. Adds `0xAlice -> sessionId` and `0xBob -> sessionId` to the authorization map.

State after step 1: session budget = 50,000 Gwei for the current subsidy epoch, authorized = `[0xAlice, 0xBob]`.

**Step 2: Attach a second credential with a Session Proof.**

Later, the user proves the Phone credential against the same `sessionId` and calls `registerCredential(sessionProof)`. The component:

1. Verifies the Session Proof via `verifySession(...)`.
2. Attaches Phone (`0x02`) to the existing session.
3. Accrues the current subsidy epoch for Phone if it is valid for that epoch.

State after step 2: session budget = 60,000 Gwei for the current subsidy epoch, authorized = `[0xAlice, 0xBob]`.

**Step 3: Enter a new subsidy epoch.**

When the next subsidy epoch begins, no proof is required. On the first `getBudget(...)` or `consumeBudget(...)` call in the new epoch, the component lazily accrues:

- 50,000 Gwei for Orb if Orb is still unexpired.
- 10,000 Gwei for Phone if Phone is still unexpired.

**Step 4: Update authorized addresses with a fresh Session Proof.**

The user wants to remove `0xBob`. They generate a fresh Session Proof whose `signalHash` binds `{remove: [0xBob]}` and call `updateAuthorizedAddresses(sessionProof, [], [0xBob])`. The component:

1. Verifies the Session Proof.
2. Removes `0xBob -> sessionId` from the authorization map.
3. Leaves the session budget unchanged, except for any lazy epoch accrual that became due before the call.

State after step 4: authorized = `[0xAlice]`.

#### Gas Accounting Flow

For an incoming transaction:

1. The protocol looks up whether the sender address has an associated session in the authorization map.
2. If the address maps to more than one session, a deterministic rule selects which session's budget to use. With the new transaction type, the sender can explicitly specify which session to use.
3. Before charging gas, the component settles any missed subsidy epochs for that session and its attached credentials.
4. The budget is reduced by `gasUsed * baseFee`.

With the new transaction type, the sender can also explicitly specify whether the transaction should consume session subsidy budget or not.

### World ID KeyRing

A **World ID KeyRing** is an account whose authorized signatory set is administered by a World ID session. Keyrings no longer derive their identity from a proof nullifier. Instead, a keyring is owned by a `sessionId`, and every keyring mutation requires a fresh Session Proof verifying continuity with that session.

World ID session proofs act as the administrative key over the KeyRing. This preserves proof-based recovery and key rotation while aligning with the new `world-id-protocol` model: uniqueness proofs are one-time enrollment events, while session proofs are the reusable continuity mechanism.

Session keys are abstract signatories authorized to sign `0x1D` transactions on behalf of the keyring; they carry no intrinsic privileges beyond signing. Per-key permissions are out of scope for this WIP (see §Extensions).

KeyRings remain orthogonal to Gas Subsidies. They are not required for subsidies, but a keyring address MAY be authorized under a subsidy session like any other account.

#### Admin Proof

Every keyring mutation is authorized by a Session Proof using the proof layout from §WorldID Gas Accounting.

For keyring operations:

- `rpId = WORLD_CHAIN_RP_ID`
- `signalHash = uint256(keccak256(abi.encode(keyringNonce, update))) >> 8`

where `keyringNonce` selects a keyring under the session and `update` is the `KeyringUpdate` (§Signal Semantics).

The Session Proof's `sessionNullifier` MUST NOT be persisted. The stable ownership key is `sessionId`.

#### Keyring Address

```text
keyringSeed = keccak256(abi.encodePacked(WORLD_ID_KEYRING_DOMAIN, sessionId, keyringNonce))
addr       = bytes20(keccak256(keyringSeed))
```

Where `sessionId` is the stored World Chain session commitment and `keyringNonce` is a user-chosen selector within that session.

The address is fixed for the lifetime of the keyring. Session key rotation does NOT change the address.

#### Session Keys

```solidity
enum KeyType { Secp256k1, P256, WebAuthn, EdDSA }

struct SessionKey {
    KeyType keyType;
    bytes   keyData;
}
```

| `KeyType`   | Value  | `keyData`                                          |
| ----------- | ------ | -------------------------------------------------- |
| `Secp256k1` | `0x00` | 33 bytes — compressed pubkey                       |
| `P256`      | `0x01` | 64 bytes — uncompressed `(x, y)`                   |
| `WebAuthn`  | `0x02` | 64 bytes — uncompressed `(x, y)` (P256 / WebAuthn) |
| `EdDSA`     | `0x03` | 32 bytes — compressed pubkey (Ed25519)             |

#### Signal Semantics

```solidity
enum UpdateMode { Create, Add, Remove }

struct KeyringUpdate {
    UpdateMode   mode;
    SessionKey[] keys;
}
```

- **`Create`** — installs `keys` as the full session key set at `addr(sessionId, keyringNonce)`. MUST revert if the keyring already exists, if `keys.length == 0`, or if `keys.length > MAX_SESSION_KEYS`.
- **`Add`** — appends `keys` to the existing set. MUST revert if the keyring does not exist, if any key is already present, or if the resulting set exceeds `MAX_SESSION_KEYS`.
- **`Remove`** — removes each entry in `keys` from the existing set. MUST revert if the keyring does not exist or if any key is absent. The set MAY become empty; a keyring with zero session keys exists but cannot sign transactions.

#### Precompile Interface

```solidity
interface IWorldIDKeyRing {
    /// @notice Apply a keyring update authorized by a World ID Session Proof.
    function submitUpdate(
        SessionProof calldata proof,
        uint256 keyringNonce,
        KeyringUpdate calldata update
    ) external returns (address keyring);

    function getSessionKeys(address keyring) external view returns (SessionKey[] memory);
    function isAuthorized(address keyring, SessionKey calldata key) external view returns (bool);
    function getOwnerSession(address keyring) external view returns (uint256 sessionId);
}
```

The precompile MUST:

1. Recompute `signalHash = uint256(keccak256(abi.encode(keyringNonce, update))) >> 8`.
2. Invoke the World ID verifier via `verifySession(...)` using `proof.sessionId`, `proof.sessionNullifier`, and the recomputed `signalHash`.
3. Derive `keyring = addr(proof.sessionId, keyringNonce)`.
4. On `Create`, record `ownerSession[keyring] = proof.sessionId`.
5. On `Add` and `Remove`, require `ownerSession[keyring] == proof.sessionId`.
6. Apply `update` atomically.

#### Recovery

Recovery is the degenerate case of an `Add` or `Remove` submitted after session keys are lost or compromised. Because the admin key is the World ID session, recovery requires only a fresh Session Proof for the owning `sessionId`.

#### Timelock

A compromised WorldID authenticator could otherwise rotate session keys in a single block and drain the keyring. Keyring updates are subject to a configurable delay:

- `Create` executes immediately (no prior session keys exist to cancel).
- `Add` and `Remove` are pending until the delay elapses.
- Existing session keys MAY cancel a pending update during the delay window.
- The revocation-then-add attack (admin removes all keys, then adds malicious ones) MUST be mitigated — e.g., by delaying removals, or by allowing recently-removed keys to cancel pending adds.

Queue / cancel / execute interface additions are TBD.

#### New Transaction Type

A `0x1D` transaction is signed by a session key of a keyring and submitted directly to the mempool (no ERC-4337 bundler required).

Fields:

- Standard [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) fields (`chain_id`, `nonce`, `max_priority_fee_per_gas`, `max_fee_per_gas`, `gas_limit`, `to`, `value`, `data`, `access_list`).
- `keyring` — address of the signing keyring.
- `signature_type` — `KeyType` enum identifying the signature scheme.
- `signature_payload` — RLP-encoded signature per scheme.
- `gas_session_id` (OPTIONAL) — session whose Gas Accounting budget to consume.

Validation:

1. Recover the session public key from `(signature_type, signature_payload, signing_hash)`.
2. Assert `IWorldIDKeyRing.isAuthorized(keyring, recovered_key)`.
3. If `gas_session_id` is present, assert `IGasAccounting.isAuthorized(keyring, gas_session_id)` and route budget consumption accordingly.

`signature_payload` per `signature_type`:

| `signature_type` | Encoding                                            |
| ---------------- | --------------------------------------------------- |
| `0x00` Secp256k1 | `rlp([y_parity, r, s])`                             |
| `0x01` P256      | `rlp([r, s])`                                       |
| `0x02` WebAuthn  | `rlp([authenticator_data, client_data_json, r, s])` |
| `0x03` EdDSA     | `rlp([R, s])`                                       |

The envelope is extensible to 2D nonces, batched transactions, and native paymaster support in follow-on WIPs.

#### Extensions

- **Per-key permissions.** Session keys MAY be extended with `(expiry, spendLimit, CallScope[])` metadata enforced by the precompile during `0x1D` validation. This extends `SessionKey` and the `KeyringUpdate` signal; the Session Proof continues to bind the full permission set via `signalHash`.
- **Session rotation / unlinkability.** All keyrings under the same session are linkable to that session. A session-rotation or session-sharding scheme is deferred.
- **Bundling, 2D nonces, paymasters.** Reserved for follow-on WIPs on the `0x1D` envelope.
- **Smart Account Backwards Compatibility** Existing Smart accounts should be able to delegate authentication via World ID key ring. This simply comes down to a 4337 module upgrade allowing execution from other entrants than the EntryPoint.

## Rationale

**Keyring state as precompile.** The session key set must live in a precompile because `0x1D` signature validation happens at the protocol level and needs to read the authorized set natively. The same precompile owns the session-proof verification path, ensuring the address-to-keyset map is only mutated under a valid World ID Session Proof.

**Enrollment nullifier plus long-lived session.** `world-id-protocol` now treats ordinary uniqueness proofs as one-time artifacts and provides Session Proofs as the reusable continuity primitive. This WIP uses both properties: the enrollment Uniqueness Proof uses a fixed action and its returned nullifier is stored on chain to ensure one-time enrollment, while the long-lived control surface for subsidy and keyring state is the persisted `sessionId`.

**Gas accounting deployment flexibility.** The gas accounting component is designed with an identical interface whether deployed as a precompile or a standard contract (e.g., predeploy). As a contract with builder-level enforcement, it avoids protocol changes and allows faster iteration. As a precompile, it enables in-protocol enforcement. Starting with a contract is a pragmatic first step; migration to a precompile is straightforward if needed.

**Automatic subsidy epochs.** Subsidy epochs separate budget issuance cadence from proof cadence. A user proves a credential once to attach it to a session, and the protocol thereafter issues its configured epoch budget automatically until expiry. This avoids periodic proof refreshes while still respecting credential validity.

**Per-credential budgets.** Different credentials represent different levels of verification. Proof-of-Human from an Orb carries more weight than a phone credential. Allowing governance to configure budget amounts per `issuerSchemaId` enables fine-grained subsidy policy.

**One session, multiple credentials.** A single session can aggregate multiple credentials over time. This allows a World ID to keep one stable session identifier for address authorization and keyring control while still receiving differentiated subsidy amounts from multiple credential types.

**Recovery timelock.** Without a delay, a compromised WorldID authenticator could instantly replace all session keys and drain the account. The timelock gives existing session key holders a window to detect and cancel malicious changes.

## Backwards Compatibility

This WIP introduces a new account type and transaction type. Existing EOA and smart contract accounts are unaffected.

The gas accounting component can authorize any account type, including legacy EOAs. This means existing accounts can opt in to WorldID gas subsidies without migrating to the new account type.

The new transaction type is required for native multi-auth. Standard EIP-1559 transactions continue to work for legacy accounts. Whether 1559 transactions from authorized accounts can consume gas budgets is an open question (see optional requirements).

## Security Considerations

**Session linkability.** A `sessionId` is a stable on-chain identifier for World Chain. Subsidy updates, keyring mutations, and any authorized addresses or keyrings tied to the same session are linkable to that session. This WIP intentionally trades the unlinkability of rotating nullifiers for session continuity and automatic epoch subsidies.

**Timelock attacks.** The revocation-then-add attack vector (admin proof removes all session keys, then adds malicious ones with no keys left to cancel) must be addressed in the timelock policy design. Possible mitigations include timelocking removals or allowing recently-removed keys to cancel pending adds.

**Keyring linkability.** Keyrings are derived from `(sessionId, keyringNonce)`. Distinct keyrings under the same session are therefore linkable to that session whenever the admin operation is observed on chain. Users and wallets SHOULD treat the session as the durable privacy boundary.

**Session key deanonymization.** A session key that is also used as an EOA or added to another keyring trivially links the keyring to that external identity. Clients SHOULD generate fresh session keys per keyring.

**Session proof replay.** A Session Proof can be observed and re-submitted before the intended transaction lands. `signalHash` binding prevents swapping in a different mutation, but a griefer can still front-run an identical payload. Harm is limited to gas and ordering if the operation is idempotent or reverts on duplicates. Implementations MAY additionally bind proof submission to `msg.sender` or another caller commitment if stronger guarantees are required.

**SessionId durability.** The chain stores only `SessionId.commitment()`, while future Session Proofs require the full off-chain `SessionId` including its `oprfSeed`. Wallets MUST back up that off-chain value. Losing it prevents generating future Session Proofs even though the on-chain session state still exists.

**Enrollment duplication resistance.** Session creation relies on two constraints together: a fixed enrollment action for the initial Uniqueness Proof, and on-chain storage of the resulting uniqueness nullifier. The fixed action ensures the same World ID yields the same enrollment nullifier; storing that nullifier on chain ensures the same World ID cannot create multiple subsidy sessions by replaying or re-enrolling.

**WebAuthn challenge binding.** For `signature_type = 0x02`, the `0x1D` `signing_hash` MUST appear as the `challenge` field inside `client_data_json`; otherwise the signature is replayable across WebAuthn contexts.

**Lazy accrual complexity.** Automatic subsidy issuance shifts work from proof submission time to settlement time. Implementations SHOULD ensure missed subsidy epochs are settled in O(1) or otherwise bounded work per credential, rather than looping unboundedly over long inactivity windows.

**Gas accounting trust boundary.** The `consumeBudget` method must be access-controlled. As a precompile, it must only be callable by the protocol during transaction execution. As a contract, it must be restricted to the builder's designated caller (e.g., via `onlyOwner` or a builder-specific access control mechanism).
