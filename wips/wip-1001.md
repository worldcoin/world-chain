## Abstract

This WIP introduces native account abstraction for World Chain with three core features: (1) a **WorldID Gas Accounting** system enabling per-credential gas subsidies for verified humans via periodic nullifier-based budgets managed by a precompile, (2) **World ID KeyRings** — accounts whose session-key set is gated by a WorldID admin proof, supporting native multi-authentication (Secp256k1, P256, WebAuthn, EdDSA) without bundler infrastructure and with no sybil restriction on the number of keyrings per WorldID, and (3) WorldID-based recovery of keyring session keys with timelock protection against compromised authenticators. A new transaction type (`0x1D`) is introduced that is signed by any authorized session key of a keyring and optionally targets a specific gas budget.

A World ID KeyRing is tangential to Gas Accounting: its sole purpose is to abstract the signature scheme authenticating an account. A keyring's address is a standard 20-byte account and MAY be authorized in the Gas Accounting authorization map like any other address.



## Motivation

### WorldID Gas Accounting

World Chain needs a mechanism to subsidize gas for verified WorldID holders while preventing abuse. The following requirements drive the design:

- **Per-credential gas budget.** Different credentials (e.g., Proof-of-Human, phone, NFC passport) should grant different budget amounts.
- **Sybil resistance.** Each WorldID should only be able to claim a gas budget once per period.
- **Anonymity.** The subsidy mechanism must not de-anonymize a WorldID. If a WorldID authorizes different accounts across periods, there should be no way to link them to the same identity.
- **Periodic refresh.** The gas budget is not one-time; it refreshes periodically.
- **Gas-price-based budget.** Budget is consumed based on `gasUsed * baseFee` at time of transaction, not raw gas units or transaction count, which could easily be abused.
- **Traffic classification.** For any transaction, it must be possible to determine whether it is authorized to consume a WorldID's gas budget.
- **Multi-account.** A single WorldID should be able to authorize multiple accounts. Budget is shared across all authorized accounts.
- **Authorization management.** Authorized accounts can be added and revoked over time. Revocation immediately stops an account from consuming the budget.
- **Out-of-protocol usage.** The accounting machinery should be usable by out-of-protocol mechanisms, e.g., if gas subsidies are implemented at the builder level only.
- **Account type agnostic (optional).** A WorldID should be able to authorize any account type, including legacy EOAs and smart accounts.
- **1559 support (optional).** Gas subsidies should be usable with standard EIP-1559 transactions to enable legacy third-party wallet support.
- **Explicit budget selection (optional).** A user should be able to specify per-transaction whether to use the budget or not.

### Native Multi-Auth

- **Multiple authentication methods.** Support for passkeys, different signature schemes, multi-sig, and WorldID as authenticators.
- **Native support.** Users should be able to submit transactions directly without bundler infrastructure.
- **Key rotation.** Authenticators can be added and removed after account creation.

### WorldID Recovery

- **Proof-based recovery.** A user can recover their account by submitting a WorldID proof, as long as they control the WorldID's authenticators.
- **Recovery means updating authenticators.** Recovery updates the set of authenticators on the account, not a migration of funds to a new address.
- **Protection from compromised authenticators.** A compromised WorldID authenticator must not immediately lead to a drained account. A safety mechanism (e.g., timelock) is required.
- **Unlinkability**. A single world id should be able to enable recovery for multiple accounts. The accounts shoudl not be linkable.

### WorldID Gas Accounting

The accounting logic is implemented in a dedicated Gas Accounting component. Time is divided into **periods** (e.g., one month). Each WorldID gets a nullifier per period, and budget must be refreshed each period. There is a governance-configurable map from credentials (identified by `issuerSchemaId`) to claimable budget amounts, where each claim requires one WorldID proof.

The component maintains an authorization map from authorized account addresses to nullifiers, and a gas budget map from nullifiers to remaining budget. These maps can be used to look up whether an address is an authorized account and, if so, to query and update its remaining budget.

The gas accounting component can be deployed as either:

- **A precompile**, enabling the protocol to natively read gas accounting state during transaction validation and execution. This is required if gas subsidies are enforced at the protocol level (e.g., the EVM itself checks and deducts budget).
- **A standard contract** (e.g., a predeploy), where the builder reads contract state via `eth_call` during block construction to determine subsidy eligibility and track budget consumption. This avoids protocol-level changes and allows iterating on subsidy logic without hard forks.

The interface is identical in both cases. The choice of deployment model determines where enforcement happens (protocol vs. builder) but does not affect the accounting logic itself.

#### Gas Accounting Interface

Manages all gas accounting state:

- **Authorization map:** maps authorized account addresses to nullifiers. One account can be mapped to multiple nullifiers.
- **Gas budget map:** maps nullifiers to remaining budget in Gwei. Budget accumulates as credential proofs are submitted.
- **Credential budget configuration:** maps `issuerSchemaId` to claimable budget amount (governance-configurable per credential type).
- **Claim tracking:** tracks which `issuerSchemaId` have been claimed per nullifier per period to prevent double-claims.

Exposes methods for:

- Proof submission (verify proof, claim budget and/or update authorized addresses)
- Budget lookup (remaining budget for a nullifier)
- Budget consumption (called by the protocol during transaction execution)

```solidity
interface IGasAccounting {
    /// @notice Submit a WorldID proof to claim budget and/or update authorized addresses.
    ///         If the credential's issuerSchemaId has not been claimed for this nullifier
    ///         in the current period, the corresponding budget is added.
    ///         in the current period, the corresponding budget is added.
    ///         If authorizedAddresses is non-empty, the authorization map is updated.
    function submitProof(
        uint256 nullifier,
        uint256 issuerSchemaId,
        bytes calldata proof,
        bytes calldata signal,
        address[] calldata addAddresses,
        address[] calldata removeAddresses
    ) external;

    /// @notice Get remaining gas budget for a nullifier in the current period.
    function getBudget(uint256 nullifier) external view returns (uint256 remainingGwei);

    /// @notice Check whether an address is authorized under a given nullifier.
    function isAuthorized(address account, uint256 nullifier) external view returns (bool);

    /// @notice Get all nullifiers associated with an address.
    function getNullifiers(address account) external view returns (uint256[] memory);

    /// @notice Check whether a credential has been claimed for a nullifier this period.
    function isClaimed(uint256 nullifier, uint256 issuerSchemaId) external view returns (bool);

    /// @notice Consume budget for a nullifier. Called during tx execution (by protocol or builder).
    function consumeBudget(uint256 nullifier, uint256 gasUsed, uint256 baseFee) external;

    /// @notice Set the claimable budget amount for a credential type. Governance only.
    function setCredentialBudget(uint256 issuerSchemaId, uint256 budgetGwei) external;
}
```

##### Authorization Map

Maps authorized account addresses to nullifiers. One account can be mapped to multiple nullifiers. Authorized accounts can be added and removed throughout the entire period via `submitProof`.

##### Gas Budget Map

Maps nullifiers to remaining budget in Gwei. Budget accumulates as new credential proofs are submitted. Tracks which `issuerSchemaId` have been claimed per nullifier per period to prevent double-claims.

#### Periods and Nullifiers

The gas budget is bound to a per-period nullifier derived from a WorldID proof:

- Relying party is World Chain (e.g., RP ID `480`)
- Action is `"period_proof" || periodNumber`
- Signal contains the action (add or remove) and the list of authorized addresses

Every WorldID proof natively carries an `issuerSchemaId` (mandatory public input in the circuit). The nullifier is credential-independent: the same action produces the same nullifier regardless of which credential is used. This means all credential proofs for a given WorldID in a given period map to the same nullifier.

Period could be e.g., one month. It likely makes sense to divide the period into multiple slots with sub-budgets for each slot to prevent traffic spikes (e.g., when all users claim their monthly budget at once after a token launch).

Per-proof behavior of the precompile:

1. If the `issuerSchemaId` has not been claimed for this nullifier in the current period, claim that credential's budget.
2. If the signal contains authorized addresses, update the authorization map (add or remove).
3. Both can happen in the same proof, or either independently.

A proof with a previously-claimed credential can be resubmitted to update authorized accounts without re-claiming budget. A proof with a new credential and empty authorized accounts list can be submitted just to claim additional budget.

#### Example: Authorization and Budget Claim Flow

Assume governance has configured the following credential budgets:

| Credential | `issuerSchemaId` | Budget |
|---|---|---|
| Proof-of-Human (Orb) | `0x01` | 50,000 Gwei |
| Phone | `0x02` | 10,000 Gwei |

A user holds a WorldID with both credentials. At the start of period 7, they want to authorize two addresses (`0xAlice`, `0xBob`) and claim their full budget.

**Step 1: First proof — authorize addresses and claim PoH budget.**

The user generates a WorldID proof with:

- Action: `"period_proof" || 7`
- Credential: Proof-of-Human (`issuerSchemaId = 0x01`)
- Signal: `{action: add, addresses: [0xAlice, 0xBob]}`

They call `submitProof(nullifier, 0x01, proof, signal, [0xAlice, 0xBob], [])`. The precompile:

1. Verifies the proof.
2. `0x01` has not been claimed for this nullifier in period 7 → adds 50,000 Gwei to the budget.
3. Adds `0xAlice → nullifier` and `0xBob → nullifier` to the authorization map.

State after step 1: budget = 50,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 2: Second proof — claim phone credential budget.**

The user generates another proof with the same action (same period, so same nullifier) but a different credential:

- Credential: Phone (`issuerSchemaId = 0x02`)
- Signal: empty (no address changes)

They call `submitProof(nullifier, 0x02, proof, signal, [], [])`. The precompile:

1. Verifies the proof.
2. `0x02` has not been claimed for this nullifier in period 7 → adds 10,000 Gwei to the budget.
3. No address changes.

State after step 2: budget = 60,000 Gwei, authorized = `[0xAlice, 0xBob]`.

**Step 3 (later): Revoke an address.**

The user wants to remove `0xBob`. They resubmit a proof with any credential (e.g., PoH again):

- Signal: `{action: remove, addresses: [0xBob]}`

They call `submitProof(nullifier, 0x01, proof, signal, [], [0xBob])`. The precompile:

1. Verifies the proof.
2. `0x01` was already claimed → no budget change.
3. Removes `0xBob → nullifier` from the authorization map. `0xBob` can no longer consume this budget.

State after step 3: budget = 60,000 Gwei (minus any consumed so far), authorized = `[0xAlice]`.

#### Budget Refresh

A nullifier with its budget and authorized accounts list is active for exactly one period. At the end of the period, all nullifiers expire.

To prevent congestion from all accounts submitting refresh proofs at the start of a new period, refresh proofs for the next period can be submitted during the current period.

Note that nullifiers of the same WorldID across different periods cannot be linked. If a WorldID authorizes independent addresses in each period, it remains fully anonymous.

#### Gas Accounting Flow

For an incoming transaction:

1. The protocol looks up whether the sender address has an associated nullifier in the authorization map.
2. If the address maps to more than one nullifier, a deterministic rule selects which nullifier's budget to use. With the new transaction type, the sender can explicitly specify which nullifier to use.
3. The budget is updated based on `gasUsed * baseFee` of the transaction.

With the new transaction type, the sender can also explicitly specify whether the transaction should consume budget or not.

### World ID KeyRing

A **World ID KeyRing** is an account whose authorized signatory set is gated by a WorldID admin proof. The keyring address derives from an admin nullifier, which itself is a deterministic function of the owning WorldID identity and a user-chosen `nonce`. A single WorldID MAY manage any number of keyrings by varying `nonce`; keyring addresses produced by the same WorldID are unlinkable by the ZK property of the OPRF nullifier proof.

The admin key over a keyring is the WorldID proof itself — not any signing key. Session keys are abstract signatories authorized to sign `0x1D` transactions on behalf of the keyring; they carry no intrinsic privileges beyond signing. Per-key permissions are out of scope for this WIP (see §Extensions).

KeyRings are orthogonal to Gas Accounting. A keyring's address MAY be authorized in the Gas Accounting map (§WorldID Gas Accounting) like any other address; the two precompiles share no state.

#### Constants

| Name                          | Value                                          | Description                                                   |
| ----------------------------- | ---------------------------------------------- | ------------------------------------------------------------- |
| `WORLD_ID_KEYRING_PRECOMPILE` | `0x000000000000000000000000000000000000001D`   | Stateful precompile managing the keyring registry             |
| `WORLD_ID_KEYRING_TAG`        | `"WORLD_ID_KEYRING"` (UTF-8, 16 bytes)         | Domain tag for keyring admin proof actions                    |
| `WORLD_CHAIN_RP_ID`           | `480`                                          | RP ID used for keyring admin proofs                           |
| `MAX_SESSION_KEYS`            | `20`                                           | Maximum authorized session keys per keyring                   |
| `WORLD_TX_TYPE`               | `0x1D`                                         | [EIP-2718](https://eips.ethereum.org/EIPS/eip-2718) tx type   |

#### Admin Proof

Every keyring mutation is authorized by a single WorldID OPRF nullifier proof parameterized as:

- `rpId = WORLD_CHAIN_RP_ID`
- `action = keccak256(abi.encodePacked(WORLD_ID_KEYRING_TAG, uint256(nonce)))`
- `signalHash = uint256(keccak256(abi.encode(update))) >> 8`

where `update` is the `KeyringUpdate` (§Signal Semantics). By circuit guarantee **(G4)** — nullifier determinism per `(identity, rpId, action)` — every admin proof against the same keyring yields the same nullifier `ν_k`, regardless of which credential is used. By **(G5)** the proof is bound to `signalHash`, so the submitted update MUST match what was committed.

The admin nullifier is **NOT** marked spent. A keyring accepts any number of admin proofs over its lifetime — a single WorldID may freely create, add to, remove from, or cease interacting with keyrings at arbitrary nonces.

#### Keyring Address

```
addr(ν_k) = bytes20(keccak256(ν_k))
```

The address is fixed for the lifetime of the keyring. Session key rotation does NOT change the address.

#### Session Keys

```solidity
enum KeyType { Secp256k1, P256, WebAuthn, EdDSA }

struct SessionKey {
    KeyType keyType;
    bytes   keyData;
}
```

| `KeyType`   | Value  | `keyData`                                          |
| ----------- | ------ | -------------------------------------------------- |
| `Secp256k1` | `0x00` | 33 bytes — compressed pubkey                       |
| `P256`      | `0x01` | 64 bytes — uncompressed `(x, y)`                   |
| `WebAuthn`  | `0x02` | 64 bytes — uncompressed `(x, y)` (P256 / WebAuthn) |
| `EdDSA`     | `0x03` | 32 bytes — compressed pubkey (Ed25519)             |

#### Signal Semantics

```solidity
enum UpdateMode { Create, Add, Remove }

struct KeyringUpdate {
    UpdateMode   mode;
    SessionKey[] keys;
}
```

- **`Create`** — installs `keys` as the full session key set at `addr(ν_k)`. MUST revert if the keyring already exists, if `keys.length == 0`, or if `keys.length > MAX_SESSION_KEYS`.
- **`Add`** — appends `keys` to the existing set. MUST revert if the keyring does not exist, if any key is already present, or if the resulting set exceeds `MAX_SESSION_KEYS`.
- **`Remove`** — removes each entry in `keys` from the existing set. MUST revert if the keyring does not exist or if any key is absent. The set MAY become empty; a keyring with zero session keys exists but cannot sign transactions.

#### Precompile Interface

```solidity
interface IWorldIDKeyRing {
    /// @notice Apply a keyring update authorized by a WorldID admin proof.
    function submitUpdate(
        uint256 keyringNullifier,            // ν_k
        uint256 nonce,                       // keyring selector; recomputed into `action`
        uint64  issuerSchemaId,
        uint64  expiresAtMin,
        uint256 credentialGenesisIssuedAtMin,
        uint256[5] calldata proof,           // compressed Groth16 proof
        KeyringUpdate calldata update
    ) external returns (address keyring);

    function getSessionKeys(address keyring) external view returns (SessionKey[] memory);
    function isAuthorized(address keyring, SessionKey calldata key) external view returns (bool);
    function getKeyringNullifier(address keyring) external view returns (uint256);
}
```

The precompile MUST:

1. Recompute `action = keccak256(abi.encodePacked(WORLD_ID_KEYRING_TAG, uint256(nonce)))`.
2. Recompute `signalHash = uint256(keccak256(abi.encode(update))) >> 8`.
3. Invoke the WorldID verifier with `(keyringNullifier, action, WORLD_CHAIN_RP_ID, …, signalHash, …, proof)`.
4. Derive `keyring = bytes20(keccak256(keyringNullifier))` and apply `update` atomically.

The precompile MUST NOT enforce uniqueness across nullifiers or nonces — no sybil restriction exists on keyrings per WorldID.

#### Recovery

Recovery is the degenerate case of an `Add` or `Remove` submitted after session keys are lost or compromised. Because the admin key is the WorldID itself, recovery requires only a fresh admin proof at the keyring's `nonce`.

#### Timelock

A compromised WorldID authenticator could otherwise rotate session keys in a single block and drain the keyring. Keyring updates are subject to a configurable delay:

- `Create` executes immediately (no prior session keys exist to cancel).
- `Add` and `Remove` are pending until the delay elapses.
- Existing session keys MAY cancel a pending update during the delay window.
- The revocation-then-add attack (admin removes all keys, then adds malicious ones) MUST be mitigated — e.g., by delaying removals, or by allowing recently-removed keys to cancel pending adds.

Queue / cancel / execute interface additions are TBD.

#### New Transaction Type

A `0x1D` transaction is signed by a session key of a keyring and submitted directly to the mempool (no ERC-4337 bundler required).

Fields:

- Standard [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) fields (`chain_id`, `nonce`, `max_priority_fee_per_gas`, `max_fee_per_gas`, `gas_limit`, `to`, `value`, `data`, `access_list`).
- `keyring` — address of the signing keyring.
- `signature_type` — `KeyType` enum identifying the signature scheme.
- `signature_payload` — RLP-encoded signature per scheme.
- `gas_nullifier` (OPTIONAL) — nullifier whose Gas Accounting budget to consume.

Validation:

1. Recover the session public key from `(signature_type, signature_payload, signing_hash)`.
2. Assert `IWorldIDKeyRing.isAuthorized(keyring, recovered_key)`.
3. If `gas_nullifier` is present, assert `IGasAccounting.isAuthorized(keyring, gas_nullifier)` and route budget consumption accordingly.

`signature_payload` per `signature_type`:

| `signature_type`      | Encoding                                            |
| --------------------- | --------------------------------------------------- |
| `0x00` Secp256k1      | `rlp([y_parity, r, s])`                             |
| `0x01` P256           | `rlp([r, s])`                                       |
| `0x02` WebAuthn       | `rlp([authenticator_data, client_data_json, r, s])` |
| `0x03` EdDSA          | `rlp([R, s])`                                       |

The envelope is extensible to 2D nonces, batched transactions, and native paymaster support in follow-on WIPs.

#### Extensions

- **Per-key permissions.** Session keys MAY be extended with `(expiry, spendLimit, CallScope[])` metadata enforced by the precompile during `0x1D` validation. This extends `SessionKey` and the `KeyringUpdate` signal; the admin proof continues to bind the full permission set via **(G5)**.
- **Cross-update unlinkability within a nonce.** Updates at a single nonce share a nullifier, so a keyring's own update history is linkable to itself across time (unlinkability across nonces is preserved). A chained-nullifier scheme is deferred.
- **Bundling, 2D nonces, paymasters.** Reserved for follow-on WIPs on the `0x1D` envelope.

## Rationale

**Keyring state as precompile.** The session key set must live in a precompile because `0x1D` signature validation happens at the protocol level and needs to read the authorized set natively. The same precompile owns the admin-proof verification path, ensuring the address-to-keyset map is only mutated under a valid WorldID admin proof.

**WorldID as admin key.** Making the admin key a WorldID proof (rather than a signing key) means session key rotation and recovery converge on a single mechanism — a fresh admin proof — and no "super key" ever needs to be stored on-device. The nonce-indexed action (`keccak256("WORLD_ID_KEYRING" || nonce)`) lets a single WorldID own unlinkably many keyrings, with each keyring's address pinned for life by `addr(ν_k) = bytes20(keccak256(ν_k))`.

**Gas accounting deployment flexibility.** The gas accounting component is designed with an identical interface whether deployed as a precompile or a standard contract (e.g., predeploy). As a contract with builder-level enforcement, it avoids protocol changes and allows faster iteration. As a precompile, it enables in-protocol enforcement. Starting with a contract is a pragmatic first step; migration to a precompile is straightforward if needed.

**Nullifier-based budget tracking.** Using WorldID nullifiers for budget tracking achieves sybil resistance (one budget per WorldID per period) while preserving anonymity (nullifiers cannot be linked across periods). The credential-independent nullifier means a WorldID only needs one identity per period regardless of how many credentials it claims budget from.

**Per-credential budgets.** Different credentials represent different levels of verification. Proof-of-Human from an Orb carries more weight than a phone credential. Allowing governance to configure budget amounts per `issuerSchemaId` enables fine-grained subsidy policy.

**Periodic refresh with early submission.** Fixed periods with expiring nullifiers provide a clean budget lifecycle. Allowing early refresh prevents a thundering herd at period boundaries.

**Recovery timelock.** Without a delay, a compromised WorldID authenticator could instantly replace all session keys and drain the account. The timelock gives existing session key holders a window to detect and cancel malicious changes.

## Backwards Compatibility

This WIP introduces a new account type and transaction type. Existing EOA and smart contract accounts are unaffected.

The gas accounting component can authorize any account type, including legacy EOAs. This means existing accounts can opt in to WorldID gas subsidies without migrating to the new account type.

The new transaction type is required for native multi-auth. Standard EIP-1559 transactions continue to work for legacy accounts. Whether 1559 transactions from authorized accounts can consume gas budgets is an open question (see optional requirements).

## Security Considerations

**Anonymity.** Nullifiers are unlinkable across periods by construction. A WorldID that authorizes different addresses each period cannot be tracked. However, if the same addresses are reauthorized across periods, on-chain observers can infer continuity.

**Timelock attacks.** The revocation-then-add attack vector (admin proof removes all session keys, then adds malicious ones with no keys left to cancel) must be addressed in the timelock policy design. Possible mitigations include timelocking removals or allowing recently-removed keys to cancel pending adds.

**Keyring unlinkability.** Two keyrings owned by the same WorldID are unlinkable across distinct `nonce` values by the ZK property of the OPRF proof. Within a single nonce, all admin proofs share a nullifier by **(G4)**, so the keyring's own update history is linkable to itself — observers can tell "these two updates are on the same keyring" but cannot tell "this keyring belongs to the same WorldID as that keyring."

**Session key deanonymization.** A session key that is also used as an EOA or added to another keyring trivially links the keyring to that external identity. Clients SHOULD generate fresh session keys per keyring.

**Admin proof replay.** Because the admin nullifier is not marked spent, any party observing an admin proof in the mempool can re-submit it. Signal binding **(G5)** prevents substituting a different update, but a griefer can front-run the original submitter. Harm is limited to gas and ordering — the resulting state is identical. Implementations MAY bind the proof to `msg.sender` or a session-key-signed commitment if stronger guarantees are required.

**WebAuthn challenge binding.** For `signature_type = 0x02`, the `0x1D` `signing_hash` MUST appear as the `challenge` field inside `client_data_json`; otherwise the signature is replayable across WebAuthn contexts.

**Refresh congestion.** Even with early refresh support, a significant fraction of users may refresh near period boundaries. Sub-period slots with proportional sub-budgets can mitigate traffic spikes.

**Gas accounting trust boundary.** The `consumeBudget` method must be access-controlled. As a precompile, it must only be callable by the protocol during transaction execution. As a contract, it must be restricted to the builder's designated caller (e.g., via `onlyOwner` or a builder-specific access control mechanism).
