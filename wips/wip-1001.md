## Abstract

A *World Chain Account* is a precompile-managed account whose 20-byte address is deterministic and lifetime-stable. Each account stores a single *admin authority* initialized at creation, which gates mutation of the account's *Keyring* — the set of session keys authorized to sign on its behalf. We extend [EIP-2718](https://eips.ethereum.org/EIPS/eip-2718) with a typed transaction envelope (`0x1D`) signed by any session key in a *World Chain Account*'s keyring.

This WIP separates two concepts that were coupled in earlier drafts:

- **Creation authorization.** A one-time proof that the caller may initialize an account with the supplied admin authority, salt, and initial session-key set. Creation can be authorized by the initial admin itself or by a [World ID 4.0](https://github.com/worldcoin/world-id-protocol) Uniqueness Proof that delegates account administration to the supplied admin authority.
- **Admin authority.** The authority stored in account state after creation. It is the only authority that can add or remove session keys after creation. This WIP specifies Secp256k1, P256, and [EIP-1271](https://eips.ethereum.org/EIPS/eip-1271) admin authorities.

A World ID proof is therefore one way to create a generic World Chain Account; it is not automatically the account's ongoing keyring admin. World ID Session Proofs, biometric recovery, threshold recovery, and other recovery policies are opt-in admin-authority or extension choices, not implicit privileges granted by account creation.

## Motivation

### Computational Sustainability & World Chain UX

Enshrining smart-account authenticators as protocol-level signature verification algorithms lets us jointly optimize transaction gas, expose a simple multi-auth API over familiar authenticators ([Passkey](https://www.w3.org/TR/webauthn-3/) / [WebAuthn](https://www.w3.org/TR/webauthn-3/), [P256](https://neuromancer.sk/std/nist/P-256), [ECDSA](https://en.wikipedia.org/wiki/Elliptic_Curve_Digital_Signature_Algorithm), [EdDSA](https://en.wikipedia.org/wiki/EdDSA)), minimize confirmation time, and remove the need for bundler infrastructure.

### Pluggable Admin Authorities

A pluggable admin model lets a single account abstraction serve multiple user populations from one precompile. **Key admins** (Secp256k1 / P256) support users and integrations with existing key-management infrastructure. **EIP-1271 admins** let a Safe, multisig, policy contract, or other smart-contract signer own the account's keyring policy. **World ID creation authorization** lets a user prove humanity at account creation time while delegating ongoing administration to one of those authorities.

Sharing a single keyring + transaction surface (`0x1D`, session keys, signal binding, timelock) across instantiations means session-key UX and integration surface are uniform regardless of how the account was created or which admin authority was initialized. A new admin instantiation reduces to specifying one verification primitive and one `AdminAuthority` encoding.

### Abstract Account Model

An *account* is created by a creation-authorized `Create` operation that fixes an initial *admin authority* `A`. After creation, `A` is immutable for the lifetime of the account unless a future admin-rotation extension is adopted. The keyring is mutated by admin-authorized `Add` or `Remove` operations. All creation and admin authorizations bind the operation to a uniform `signalHash` defined below; replay protection across updates is provided by a per-account monotonic `adminNonce`.

Creation authorization and admin authorization are deliberately separate. A `CREATE_AUTH_WORLD_ID` proof can authorize account creation and delegate administration to an ECDSA key, P256 key, or EIP-1271 contract, but it does not grant the World ID holder ongoing keyring mutation rights unless the stored admin authority explicitly implements such rights in a future extension.

#### Account State

Per-account state stored in the precompile:

```
struct AdminAuthority {
    uint8 adminAuthType;  // ADMIN_AUTH_*
    bytes authData;       // variable length per admin authority type
}

struct Account {
    AdminAuthority adminAuthority;  // fixed at creation
    SessionKey[]   sessionKeys;     // see Session Keys
    uint64         adminNonce;      // monotonic; replay protection
    // pendingUpdates: see Timelock
}
```

`adminAuthority` is the persistent public material and verification type used to verify subsequent updates. Its `authData` encoding is per-instantiation (e.g., a compressed pubkey for Secp256k1 admin, or a contract address for EIP-1271 admin). It is set at creation and is not directly mutable by this WIP — see [Extensions](#extensions) for admin rotation.

#### Admin Authority Interface

The abstract operation each admin authority provides is:

```
verifyAdmin(signalHash, adminAuthorization, adminAuthority) → bool
```

`adminAuthorization` and `adminAuthority.authData` are byte strings whose encoding is per-instantiation. The precompile invokes this operation during `update`, and during `create` when `creationAuthType == CREATE_AUTH_ADMIN`. Each instantiation specifies how the operation is realized — see [Admin Authority Instantiations](#admin-authority-instantiations).

Creation authorization is a separate operation:

```
verifyCreate(signalHash, creationAuthorization, accountSalt, initialAdminAuthority) → bool
```

For `CREATE_AUTH_ADMIN`, `verifyCreate` is exactly `verifyAdmin(signalHash, creationAuthorization, initialAdminAuthority)`: the initial admin signer authorizes its own account initialization. For `CREATE_AUTH_WORLD_ID`, `verifyCreate` verifies a World ID Uniqueness Proof whose signal is the same `Create` `signalHash`, consumes the proof's creation nullifier, and stores only `initialAdminAuthority` in account state.

#### Address Derivation

```
adminAuthorityHash = keccak256(abi.encode(adminAuthority))
addr = bytes20(keccak256(abi.encodePacked(ACCOUNT_ADDRESS_DOMAIN, accountSalt, adminAuthorityHash)))
```

`accountSalt` is a 32-byte caller-chosen value. `adminAuthorityHash` commits to the initial admin authority type and public verification material. An admin authority can control multiple accounts by using distinct salts. The account address is independent of `creationAuthType`, so the account is the same generic account whether creation was authorized by the admin itself or by a World ID proof delegating to that admin.

The address is fixed for the lifetime of the account; session-key rotation does **NOT** change the address.

#### Session Keys

```solidity
enum KeyType { Secp256k1, P256, WebAuthn, EdDSA }

struct SessionKey {
    KeyType keyType;
    bytes   keyData;
}
```

| `KeyType`   | Value  | Elliptic Curve                                                                          | `keyData`                                                                                   |
| ----------- | ------ | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| `Secp256k1` | `0x00` | [secp256k1](https://en.bitcoin.it/wiki/Secp256k1)                                       | 33 bytes — compressed public key                                                            |
| `P256`      | `0x01` | [secp256r1](https://neuromancer.sk/std/nist/P-256) (NIST P-256)                         | 64 bytes — uncompressed `(x, y)`                                                            |
| `WebAuthn`  | `0x02` | [secp256r1](https://neuromancer.sk/std/nist/P-256) (NIST P-256)                         | 64 bytes — uncompressed `(x, y)` (P256 under [WebAuthn](https://www.w3.org/TR/webauthn-3/)) |
| `EdDSA`     | `0x03` | [edwards25519](https://datatracker.ietf.org/doc/html/rfc8032#section-5.1)               | 32 bytes — compressed public key (Ed25519)                                                  |

`KeyType`, `creationAuthType`, and `adminAuthType` occupy disjoint domains: their numeric values are defined in different enums and MUST NOT be used interchangeably in the precompile interface or the address-derivation preimage. Session keys carry no intrinsic privileges beyond signing `0x1D` transactions; per-key permissions (expiry, spend limits, call scopes) are out of scope for this WIP (see [Extensions](#extensions)).

#### Signal Semantics

```solidity
enum UpdateMode { Create, Add, Remove }

struct KeyringUpdate {
    UpdateMode   mode;
    SessionKey[] keys;
}
```

- **`Create`** — installs `keys` as the full keyring at the account's address and initializes the account with the supplied admin authority. MUST revert if the account already exists, if `keys.length == 0`, or if `keys.length > MAX_SESSION_KEYS`.
- **`Add`** — appends `keys` to the existing keyring. MUST revert if the account does not exist, if any key is already present, or if the resulting keyring would exceed `MAX_SESSION_KEYS`.
- **`Remove`** — removes each entry in `keys` from the existing keyring. MUST revert if the account does not exist or if any key is absent. The keyring MAY become empty; an account with zero session keys exists but cannot sign transactions until the next `Add`.

The `signalHash` an admin authority binds to is uniform across instantiations.

```solidity
struct CreateSignal {
    bytes32       tag;             // == keccak256("WORLD_CHAIN_ACCOUNT_CREATE")
    uint8         creationAuthType;
    address       msgSender;
    bytes32       accountSalt;
    AdminAuthority adminAuthority;
    KeyringUpdate keyringUpdate;
}

struct UpdateSignal {
    bytes32       tag;             // == keccak256("WORLD_CHAIN_ACCOUNT_UPDATE")
    address       accountAddress;
    uint64        adminNonce;
    address       msgSender;
    KeyringUpdate keyringUpdate;
}
```

```
signalHash = uint256(keccak256(abi.encode(signal))) >> 8
```

where `signal` is a populated `CreateSignal` for `Create` and an `UpdateSignal` for `Add` / `Remove`.

`msgSender` is the EVM `msg.sender` of the precompile call — binding it prevents an in-flight authorization observed in the mempool from being replayed by a different submitter. For `Create`, `accountAddress` is omitted because it is derived from `accountSalt` and `adminAuthority`; the domain tag, `creationAuthType`, `msgSender`, `accountSalt`, `adminAuthority`, and keyring payload together bind the authorization to the exact initialization being requested. For updates, `accountAddress` distinguishes admin authorizations across accounts the same admin might control on different salts or domains.

The `>> 8` truncation reduces the digest to 248 bits so it fits within the BN254 scalar field (matching the WorldID 4.0 proof system's signal field width). Truncation is uniform across instantiations even when the verification primitive does not require the BN254 fit, so the same `signalHash` value is canonical across all creation authorization and admin authority types.

#### Replay Protection

The precompile stores a per-account monotonic `adminNonce`, initialized to `0` at creation and incremented by `1` on every successful `Add` or `Remove`. Admin authorizations for updates are verified against `signalHash` containing the current value of `adminNonce`; an admin authorization that was valid for `adminNonce = n` is no longer valid once the precompile has incremented past `n`.

For `Create`, replay is precluded by account-existence: the precompile MUST revert if the account address is already occupied. A creation authorization can only be applied once because the post-condition (account exists at `addr`) blocks any subsequent attempt at the same address.

For `CREATE_AUTH_WORLD_ID`, the precompile additionally maintains a consumed-nullifier set keyed by the World ID creation nullifier. The precompile MUST reject creation if the supplied `worldIDAccountNullifier` has already been consumed, even if the target account address is different. This prevents one World ID creation action from initializing multiple generic accounts.

This is the abstract replay-protection contract; each instantiation realizes it via its own verification primitive. Instantiations MUST NOT introduce additional bespoke replay machinery unless their authorization primitive needs one-time state, as World ID creation authorization does with consumed nullifiers.

#### Timelock

A compromised admin authority could otherwise rotate session keys in a single block and drain an account. Account updates are subject to a configurable delay:

- `Create` executes immediately (no prior session keys exist to cancel).
- `Add` and `Remove` are pending until the delay elapses.
- Existing session keys MAY cancel a pending update during the delay window.
- The revocation-then-add attack (an admin-authorized update removes all keys, then adds malicious ones with no incumbents left to cancel) MUST be mitigated — e.g., by delaying removals, or by allowing recently-removed keys to cancel pending adds.

Queue / cancel / execute interface additions are TBD. Timelock semantics apply uniformly across admin authority instantiations.

#### Precompile Interface

```solidity
interface IWorldChainAccount {
    struct AdminAuthority {
        uint8 adminAuthType;
        bytes authData;
    }

    /// @notice Create an account. `creationAuthorization` layout is creation-auth-type-specific.
    function create(
        uint8 creationAuthType,
        bytes32 accountSalt,
        AdminAuthority calldata adminAuthority,
        bytes calldata creationAuthorization,
        KeyringUpdate calldata keyringUpdate          // mode == Create
    ) external returns (address account);

    /// @notice Apply an Add/Remove update. `adminAuthorization` layout is admin-authority-specific.
    function update(
        address account,
        KeyringUpdate calldata keyringUpdate,         // mode != Create
        bytes calldata adminAuthorization
    ) external;

    // Reads
    function getAdminAuthority(address account) external view returns (AdminAuthority memory);
    function getAdminNonce(address account) external view returns (uint64);
    function getSessionKeys(address account) external view returns (SessionKey[] memory);
    function isAuthorized(address account, SessionKey calldata key) external view returns (bool);
}
```

`creationAuthorization` and `adminAuthorization` are opaque bytes blobs. Each creation authorization instantiation specifies the ABI-encoded tuple decoded by `create`; each admin authority instantiation specifies the byte encoding decoded by `update`.

For `create`, the precompile MUST:

1. Require `keyringUpdate.mode == Create`.
2. Decode and validate `adminAuthority` per its `adminAuthType`.
3. Recompute the `Create` `signalHash` over `(creationAuthType, msg.sender, accountSalt, adminAuthority, keyringUpdate)`.
4. Verify `creationAuthorization` per `creationAuthType` against the `Create` `signalHash`, `accountSalt`, and `adminAuthority`.
5. Derive `accountAddress = bytes20(keccak256(abi.encodePacked(ACCOUNT_ADDRESS_DOMAIN, accountSalt, keccak256(abi.encode(adminAuthority)))))`.
6. Require the account does not already exist at `accountAddress`.
7. Initialize the account: store `adminAuthority`, `sessionKeys = keyringUpdate.keys`, `adminNonce = 0`.

For `update`, the precompile MUST:

1. Require `keyringUpdate.mode != Create`.
2. Load the account's stored `adminAuthority` and `adminNonce`.
3. Recompute the `Update` `signalHash` over `(accountAddress, adminNonce, msg.sender, keyringUpdate)`.
4. Verify `adminAuthorization` per `adminAuthority.adminAuthType` against `signalHash` and `adminAuthority`.
5. Increment `adminNonce`.
6. Apply `keyringUpdate`, queued behind the timelock.

> **Note — Precompile gas accounting out of scope.** The specific gas pricing schedule for interacting with `WORLD_CHAIN_ACCOUNT_PRECOMPILE` — covering creation verification cost (Groth16 / ECDSA / P256 / EIP-1271), admin verification cost, per-session-key storage mutation, timelock queue/cancel/execute operations, and the amortized cost of `isAuthorized` reads during `0x1D` validation — is **NOT** defined here. A follow-on revision MUST specify gas costs per operation per creation authorization and admin authority type.

### Creation Authorization Instantiations

Each creation authorization instantiation specifies:

1. The `creationAuthType` discriminator.
2. The `creationAuthorization` layout for `create()`.
3. The verification primitive.
4. Any one-time replay state maintained outside account state.

Creation authorization is one-time. After the account is initialized, the stored `adminAuthority` is authoritative for all keyring mutations.

#### Admin-Authorized Creation

`creationAuthType = CREATE_AUTH_ADMIN = 0x00`.

`creationAuthorization` is encoded exactly like an `adminAuthorization` for the supplied `adminAuthority`, but it signs the `Create` `signalHash` instead of an `Update` `signalHash`.

Creation flow:

1. Decode and validate `adminAuthority`.
2. Recompute the `Create` `signalHash` over `(CREATE_AUTH_ADMIN, msg.sender, accountSalt, adminAuthority, keyringUpdate)`.
3. Invoke `verifyAdmin(signalHash, creationAuthorization, adminAuthority)`.
4. Derive `addr` from `(accountSalt, adminAuthority)` per [Address Derivation](#address-derivation).
5. Require account does not exist at `addr`.
6. Store `adminAuthority`, `sessionKeys = keyringUpdate.keys`, `adminNonce = 0`.

This path lets a generic public key or EIP-1271 signer initialize a World Chain Account without any World ID proof.

#### World ID Creation Authorization

`creationAuthType = CREATE_AUTH_WORLD_ID = 0x01`.

Per-instantiation constants:

| Name                          | Value                                        | Description                                           |
| ----------------------------- | -------------------------------------------- | ----------------------------------------------------- |
| `WORLD_CHAIN_RP_ID`           | `480`                                        | World Chain's registered RP ID                        |
| `WORLD_ID_ACCOUNT_TAG`        | `"WORLD_ID_ACCOUNT"` (UTF-8, 16 bytes)       | Domain tag for World ID account creation actions      |

`creationAuthorization` layout (conceptual; ABI-encoded as a tuple):

```
WorldIDCreationAuthorization {
    uint256       worldIDAccountNullifier;      // ν_k (public input to verify and consume)
    uint256       worldIDAccountNonce;          // recomputed into action
    uint256       proofNonce;                   // verifier request nonce
    uint64        issuerSchemaId;
    uint64        expiresAtMin;
    uint256       credentialGenesisIssuedAtMin;
    uint256[5]    proof;                        // compressed Groth16 proof
}
```

The Uniqueness Proof is parameterized by:

- `rpId = WORLD_CHAIN_RP_ID`
- `action = keccak256(abi.encodePacked(WORLD_ID_ACCOUNT_TAG, uint256(worldIDAccountNonce)))`
- `signalHash` per the abstract `Create` formula

Creation flow:

1. Decode `creationAuthorization`.
2. Compute `action`.
3. Recompute the `Create` `signalHash` over `(CREATE_AUTH_WORLD_ID, msg.sender, accountSalt, adminAuthority, keyringUpdate)`.
4. Require `worldIDAccountNullifier` has not already been consumed by this precompile.
5. Invoke `WorldID.verify(worldIDAccountNullifier, action, WORLD_CHAIN_RP_ID, proofNonce, signalHash, issuerSchemaId, expiresAtMin, credentialGenesisIssuedAtMin, proof)`. The verifier asserts the proof commits to the supplied `(ν_k, action, signalHash)` tuple under the relying party's public inputs.
6. Derive `addr` from `(accountSalt, adminAuthority)` per [Address Derivation](#address-derivation).
7. Require account does not exist at `addr`.
8. Mark `worldIDAccountNullifier` consumed.
9. Store `adminAuthority`, `sessionKeys = keyringUpdate.keys`, `adminNonce = 0`.

The World ID proof delegates account administration to the supplied `adminAuthority`; it does **NOT** become the account's stored admin authority. The creation nullifier `ν_k` is retained only in the precompile's consumed-nullifier set. It is not part of the account address and is not consulted for future keyring updates.

A single World ID `MAY` create any number of accounts by varying `worldIDAccountNonce` and choosing an `accountSalt` for each creation; accounts created under distinct `worldIDAccountNonce` values remain unlinkable to each other by the ZK property of the OPRF proof, excluding linkage through the user-chosen admin authority, session keys, or application behavior.

##### OPRF Nullifier Proof Primer

An `OPRFNullifierProof` is a [Groth16](https://eprint.iacr.org/2016/260) proof over [BN254](https://hackmd.io/@jpw/bn254) submitted on-chain as `uint256[5] = [a, b₀, b₁, c, merkle_root]`. The `WorldIDVerifier` reconstructs the 15 public inputs from calldata and on-chain registry lookups (notably `merkle_root` via `WorldIDRegistry.isValidRoot()` and `oprfPublicKey` via `OprfKeyRegistry[rpId]`) and invokes the Groth16 verifier.

The circuit proves knowledge of a private witness $w = (s, k, p, \beta, R_{\mathrm{oprf}}, \mathit{cred}, \sigma_{\mathit{cred}})$ satisfying:

- **(G1) Merkle membership.** The authenticator key $k$ is a leaf in the `WorldIDRegistry` tree at the declared root.
- **(G2) Credential validity.** $\mathit{cred}$ bears a valid EdDSA-Poseidon2 signature over [BabyJubJub](https://eips.ethereum.org/EIPS/eip-2494) from the issuer identified by `issuerSchemaId`, and has not expired.
- **(G3) OPRF correctness.** The OPRF response $R_{\mathrm{oprf}}$ was evaluated under the registered key for `rpId`, with a DLog equality proof binding blinded and unblinded responses.
- **(G4) Deterministic nullifier derivation.**

$$q = \mathrm{Poseidon2}(\text{"World ID Query"}, \; i, \; \mathrm{rpId}, \; a)$$

$$\nu = \mathrm{Poseidon2}(\text{"World ID Proof"}, \; q, \; R_{\mathrm{oprf}}.x, \; R_{\mathrm{oprf}}.y)$$

  $\nu$ is deterministic per $(i, \mathrm{rpId}, a)$: the same identity proving under the same $(\mathrm{rpId}, a)$ always yields the same $\nu$, regardless of blinding factor $\beta$ or proof randomness.

- **(G5) Signal binding.** The `signalHash` is "dummy-squared" in the circuit (constrained to $h \cdot h = h^2$), preventing a proof from being reused with a different signal.

For account creation, **(G4)** binds $\nu_k$ to the triple (owner leaf, `WORLD_CHAIN_RP_ID`, `action(worldIDAccountNonce)`) and **(G5)** binds the Uniqueness Proof to the exact `Create` payload it authorizes, including `creationAuthType`, `accountSalt`, `adminAuthority`, and the initial keyring.

### Admin Authority Instantiations

Each admin authority instantiation specifies:

1. The `adminAuthority.authData` encoding.
2. The `adminAuthorization` layout for `update()`.
3. The verification primitive.
4. Recovery semantics.

Future instantiations (EdDSA, WebAuthn-as-admin, World ID Session-Proof admin, multi-sig) slot in as additional sub-sections without changing the abstract account model.

#### Secp256k1 Admin

`adminAuthority.adminAuthType = ADMIN_AUTH_SECP256K1 = 0x00`.

`adminAuthority.authData` is the 33-byte compressed secp256k1 admin public key.

`adminAuthorization` is the byte-encoding of a single secp256k1 ECDSA signature `(r, s, v)` over the relevant `signalHash`.

Verification flow:

1. Decode the admin public key from `adminAuthority.authData`.
2. Verify the signature is valid for `(signalHash, adminPubkey)`.
3. Implementations MUST enforce low-`s` to reject malleable signatures.

When used with `CREATE_AUTH_ADMIN`, the signature is over the `Create` `signalHash`. For updates, the signature is over the `Update` `signalHash`.

##### Recovery

**Not supported by the precompile.** Loss of the secp256k1 admin key is terminal unless the account has adopted a future admin-rotation or recovery extension.

#### P256 Admin

`adminAuthority.adminAuthType = ADMIN_AUTH_P256 = 0x01`.

`adminAuthority.authData` is the 64-byte uncompressed P256 (`secp256r1`) admin public key, encoding `(x, y)` as 32-byte big-endian coordinates concatenated.

`adminAuthorization` is the byte-encoding of a P256 ECDSA signature `(r, s)` over the relevant `signalHash`.

Verification flow:

1. Decode the admin public key from `adminAuthority.authData`.
2. Verify the signature is valid for `(signalHash, adminPubkey)`, via the [RIP-7212](https://github.com/ethereum/RIPs/blob/master/RIPS/rip-7212.md) precompile or equivalent host function.
3. Implementations MUST reject off-curve points and enforce low-`s`.

When used with `CREATE_AUTH_ADMIN`, the signature is over the `Create` `signalHash`. For updates, the signature is over the `Update` `signalHash`.

##### Recovery

**Not supported by the precompile.** Same warning as Secp256k1 admin.

#### EIP-1271 Admin

`adminAuthority.adminAuthType = ADMIN_AUTH_ERC1271 = 0x02`.

`adminAuthority.authData` is `abi.encode(address signerContract)`.

`adminAuthorization` is an opaque bytes payload forwarded to `signerContract.isValidSignature(bytes32 signalHash, bytes adminAuthorization)`.

Verification flow:

1. Decode `signerContract` from `adminAuthority.authData`.
2. Require `signerContract` contains code at verification time.
3. Invoke `isValidSignature(signalHash, adminAuthorization)` using `STATICCALL`.
4. Require the return value equals the EIP-1271 magic value `0x1626ba7e`.

An EIP-1271 admin lets a Safe, multisig, policy contract, or other contract signer define its own threshold, recovery, module, or delay policy. The precompile treats the contract's magic-value response as the complete admin authorization decision.

##### Recovery

Recovery semantics are defined by the EIP-1271 signer contract. The precompile does not add additional recovery powers.

### Recovery

Recovery is not a separate primitive in this WIP. Lost or compromised session keys are recovered by submitting an admin-authorized `Add` or `Remove`, subject to the timelock. If the stored admin authority is a raw Secp256k1 or P256 key, loss of that admin key is terminal. If the stored admin authority is an EIP-1271 contract, recovery is whatever the contract implements.

A World ID proof used for account creation has no post-creation recovery privilege by itself. World ID Session Proof recovery, biometric recovery, threshold recovery, or a long-delay World ID escape hatch MAY be added as future admin authority or recovery-module extensions, but must be explicitly initialized and visible in account state.

### Transaction Type `0x1D`

A `0x1D` transaction is signed by a session key of a World Chain Account and submitted directly to the mempool (no [ERC-4337](https://eips.ethereum.org/EIPS/eip-4337) bundler required). `0x1D` transactions are admin-authority-agnostic: they are signed by session keys regardless of which admin authority controls the account, and the validation path reads only the account's keyring.

#### Envelope

```
0x1D || rlp([
    chain_id,                    // 0
    nonce,                       // 1
    max_priority_fee_per_gas,    // 2
    max_fee_per_gas,             // 3
    gas_limit,                   // 4
    to,                          // 5
    value,                       // 6
    data,                        // 7
    access_list,                 // 8
    account,                     // 9   ← signer's World Chain Account address
    signature_type,              // 10
    session_key,                 // 11  ← `keyData` of the signing session key (per §Session Keys)
    signature_payload,           // 12
])
```

```
signing_hash = keccak256(0x1D || rlp(fields[0..=11]))
```

The `signing_hash` covers every field except `signature_payload` itself, binding the declared `signature_type` and `session_key` to the signature and ruling out scheme-confusion or pubkey-substitution attacks at the consensus layer.

`session_key` is the `keyData` bytes of the `SessionKey` (§Session Keys) used to authenticate the transaction. Its `KeyType` is implied by `signature_type` — implementations MUST reject any transaction where the length of `session_key` does not match the byte length specified for the corresponding `KeyType`.

| `signature_type`                                         | `signature_payload`                                 |
| -------------------------------------------------------- | --------------------------------------------------- |
| `0x00` [Secp256k1](https://en.bitcoin.it/wiki/Secp256k1) | `rlp([y_parity, r, s])`                             |
| `0x01` [P256](https://neuromancer.sk/std/nist/P-256)     | `rlp([r, s])`                                       |
| `0x02` [WebAuthn](https://www.w3.org/TR/webauthn-3/)     | `rlp([authenticator_data, client_data_json, r, s])` |
| `0x03` [EdDSA](https://en.wikipedia.org/wiki/EdDSA)      | `rlp([R, s])`                                       |

For `Secp256k1`, the `session_key` field is redundant with the recoverable public key; implementations MUST verify the recovered key matches the declared `session_key` (compressed-form encoding) before accepting the transaction.

#### Validation

1. Verify `signature_payload` against `signing_hash` using `session_key` as the verifying key (for `Secp256k1`, equivalently: recover and assert equality with `session_key`).
2. Assert `IWorldChainAccount.isAuthorized(account, SessionKey { signature_type, session_key })`.

The envelope is extensible to 2D nonces, batched transactions, and native paymaster support in follow-on WIPs.

### Extensions

- **Per-key permissions.** Session keys MAY be extended with `(expiry, spendLimit, CallScope[])` metadata enforced by the precompile during `0x1D` validation. This extends `SessionKey` and the `KeyringUpdate` signal; the admin authorization continues to bind the full permission set via `signalHash`.
- **Admin rotation.** Replacing an account's stored `adminAuthority` under admin authorization, with timelock. Deferred — the admin authority is pinned at creation in this WIP.
- **Multi-admin.** M-of-N admin authorization for a single account. Deferred as native precompile state; it can already be modeled by an EIP-1271 admin contract.
- **Additional admin authority instantiations.** EdDSA-admin, WebAuthn-as-admin, World ID Session-Proof admin, and native multisig admin.
- **World ID recovery module.** A Secp256k1 / P256 / EIP-1271 admin account MAY explicitly enroll a World ID as a recovery-only authority that, after a long timelock, can rotate the admin authority or session keyring. This MUST be visible in account state and MUST NOT be implied by a creation proof.
- **World ID credential attachment.** A generic account MAY later attach a World ID proof for subsidy, reputation, or application authorization systems without changing the stored account admin.
- **Bundling, 2D nonces, paymasters.** Reserved for follow-on WIPs on the `0x1D` envelope.

## Rationale

**Account state as precompile.** The keyring lives in a precompile because `0x1D` signature validation happens at the protocol level and must read the authorized set natively. The same precompile owns the admin verification paths, ensuring the address-to-keyset map is only mutated under valid admin authorizations.

**Creation authorization separate from administration.** A World ID proof can establish that a human authorized account initialization, but that does not require World ID Session Proofs or biometric recovery to become the root of trust for all future keyring mutations. Delegating to an explicit admin authority at creation keeps generic account abstraction intact and makes recovery assumptions visible.

**Mutable session keyring.** Session keys are operational keys, not recovery roots. Fixing them permanently at account creation makes loss of any session key a permanent account compromise. A timelocked admin mutation path lets compromised session keys be removed while still giving existing session keys a cancellation window against malicious admin updates.

**Pluggable admin authorities.** Decoupling the keyring + transaction surface from the admin verification primitive lets a single account abstraction serve users with hardware-secured root keys, World ID creation proofs, and integrations driven by existing contract signers. Sharing `0x1D`, session keys, signal binding, and timelock across instantiations means session-key UX and integration surface are uniform regardless of creation path or admin authority type. Each new admin instantiation reduces to one verification primitive plus one `AdminAuthority` encoding.

**Single admin authority.** Each account has exactly one admin authority. M-of-N admin authorization is a useful generalization but is deferred — it adds quorum-management state and protocol surface that this WIP does not require for its initial use cases. Multi-admin can be added as an Extension without changing the abstract account model.

**Admin authority pinned at creation.** Address determinism: `accountSalt` and the initial `adminAuthorityHash` are part of the address preimage, so changing the admin authority mid-lifetime would change the address unless a future rotation extension explicitly decouples rotation from address derivation.

**No implicit World ID recovery.** Biometric recovery is opt-in and has distinct trust assumptions. A World ID creation proof does not enroll biometric recovery, does not store a `sessionId`, and does not grant future keyring mutation rights. Users requiring recoverability choose an EIP-1271 admin with its own recovery policy or a future explicit recovery extension.

**Uniform replay protection.** All admin authorizations bind to a `signalHash` containing a per-account monotonic `adminNonce`; the precompile increments `adminNonce` on every successful update and rejects authorizations against stale values. Across instantiations, this gives one replay-protection contract — no instantiation-specific bespoke replay machinery, and no ambiguity around what "fresh" means per instantiation.

**Generic address derivation.** Deriving the account address from `ACCOUNT_ADDRESS_DOMAIN`, `accountSalt`, and `adminAuthorityHash` makes the account independent of how it was created. A World ID proof, an admin signature, or a contract signer can all initialize the same generic account shape.

**`msg.sender` binding in `signalHash`.** Including `msg.sender` in both `Create` and `Update` `signalHash` definitions kills cross-submitter replay of in-flight authorizations: a witnessed authorization with `msg.sender = X` will not verify if resubmitted by a different account. Self-replay by `X` is a no-op (the resulting state is identical). This subsumes the `MAY bind proofs to msg.sender` mitigation noted in earlier drafts of this WIP.

**World ID proofs as creation authorization.** A World ID Uniqueness Proof binds the `Create` signal, including the chosen admin authority and initial session keys. The consumed-nullifier set prevents reusing the same World ID account action to create multiple accounts, while varying `worldIDAccountNonce` lets one World ID initialize multiple unlinkable accounts when desired. `accountSalt` selects the deterministic address for each delegated admin authority.

**No sybil constraint on accounts.** Unlinkability across `worldIDAccountNonce` values is a stronger property than one-account-per-identity, and sybil resistance at the account layer is not the right locus — it belongs to whatever system consumes account addresses (e.g., gas subsidy accounting, application-layer rate limits), which can bind its own per-WorldID limits via independent nullifier domains.

**Recovery timelock.** Without a delay, a compromised admin authority could instantly replace all session keys and drain the account. The timelock gives existing session key holders a window to detect and cancel malicious changes. The same timelock applies regardless of admin authority type.

## Backwards Compatibility

This WIP introduces a new account type and transaction type. Existing EOAs and smart contract accounts are unaffected. Standard [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) transactions continue to work for legacy accounts; the `0x1D` envelope is required only for World-Chain-Account-authenticated execution.

This WIP is Draft. The split between creation authorization and stored admin authority, the new address derivation formula, and the expanded `Create` `signalHash` are breaking changes relative to earlier drafts of this WIP. Implementations targeting prior drafts MUST update account creation, address derivation, and `signalHash` recomputation accordingly. No production accounts exist.

### Existing Safe Accounts

Existing [Safe](https://safe.global) accounts MAY act as EIP-1271 admin authorities for World Chain Accounts. In that configuration, Safe owner and module policy controls keyring mutations through `isValidSignature`, while `0x1D` execution remains authorized by the World Chain Account's session keyring. Existing Safe accounts MAY also install a module that recognizes a World Chain Account as an execution authority without migrating assets or changing the Safe address.

## Security Considerations

### Front-Running Authorizations

Both `Create` and `Update` `signalHash` definitions bind `msg.sender`, so a witnessed authorization observed in the mempool cannot be replayed by a different submitter — its `signalHash` would not match. This applies to all instantiations. A griefer can still re-submit an authorization unchanged from the original `msg.sender`, but the resulting state is identical; harm is limited to gas and ordering.

### Timelock Attacks

The revocation-then-add attack vector (an admin authorization removes all session keys, then adds malicious ones with no keys left to cancel) MUST be addressed in the timelock policy design. Possible mitigations include timelocking removals or allowing recently-removed keys to cancel pending adds. This applies regardless of admin authority type.

### Session Key Deanonymization

A session key that is also used as an EOA, or added to a different World Chain Account, trivially links the account to that external identity. Clients SHOULD generate fresh session keys per account.

### Signature-key Ambiguity

A World Chain Account may store up to `MAX_SESSION_KEYS` session keys of mixed `KeyType`, but a `0x1D` transaction declares a single `signature_type`. The spec should define whether a key index is included in the envelope or whether the first matching key is authoritative.

### WebAuthn Challenge Binding

For session-key `signature_type = 0x02`, the `0x1D` `signing_hash` MUST appear as the `challenge` field inside `client_data_json`; otherwise the signature could be replayed from a different WebAuthn context.

### World ID Creation: Account Unlinkability

Two accounts created by the same World ID under distinct `worldIDAccountNonce` values are unlinkable by the ZK property of the OPRF proof. They can still be linked if the user reuses the same admin authority, session key, account salt pattern, or application behavior. The creation transaction publicly links the newly-created account to that proof's nullifier.

### World ID Creation: Stale Merkle Root

The `WorldIDRegistry` accepts roots within a validity window. An attacker whose credential has been revoked could race to submit `create` against a still-valid but stale root before expiry. The `expiresAtMin + minExpirationThreshold ≥ block.timestamp` check bounds this window, but its duration is a trust parameter.

### World ID Creation: OPRF Key Rotation

If the OPRF key in `OprfKeyRegistry[480]` is rotated, creation nullifier derivations change. Existing accounts are unaffected because their stored admin authority is not a World ID session, but clients and verifiers MUST define how `worldIDAccountNonce` values and consumed nullifiers are interpreted across OPRF key epochs.

### Raw-Key Admin: Admin Key Loss Is Terminal

For Secp256k1 and P256 admin instantiations, loss of the admin key permanently bricks the account unless a future explicit recovery extension has been enrolled. Wallets implementing raw-key admin accounts MUST surface this risk to users and SHOULD encourage hardware-backed key custody.

### Raw-Key Admin: Linkability

A Secp256k1 or P256 admin pubkey is recoverable from any creation transaction's calldata and is part of `adminAuthorityHash`. Reusing an admin key across accounts, salts, or chains is a deanonymization vector.

### Raw-Key Admin: No Liveness Check

A raw-key-admin account has no protocol-level liveness check on session-key-signed transactions. Session-key compromise can drain account-controlled value at the rate session keys can sign, bounded only by per-key limits (if any) and the timelock on keyring mutation. Sibling WIPs that require World ID liveness above a value threshold do NOT apply unless the account separately attaches a World ID proof or opts into a relevant extension.

### EIP-1271 Admin: Contract Risk

An EIP-1271 admin inherits the signer contract's security assumptions. Bugs, upgradeability mistakes, module compromise, or unsafe threshold policy in the signer contract can compromise keyring administration. Implementations MUST use `STATICCALL` for `isValidSignature` and MUST treat reverts or non-magic return values as failed authorization.

### World ID Creation Does Not Imply Recovery

Wallets and applications MUST NOT present a World-ID-created account as World-ID-recoverable unless a recovery authority is explicitly present in account state. The creation proof only authorizes initialization and consumes a nullifier; it is not consulted for future `Add` or `Remove` operations.
